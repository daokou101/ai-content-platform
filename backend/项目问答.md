# AI 内容创作平台 — 项目问答

## 一、项目概览

### Q: 这个项目是做什么的？
A: 这是一个 **AI 内容创作平台**。用户输入关键词、选择模板类型，后端调 DeepSeek API 生成内容，通过 SSE 流式逐字返回到前端。支持内容管理、分类、版本历史、回滚、收藏等功能。

### Q: 技术栈是什么？

| 分层 | 技术 |
|------|------|
| 后端框架 | Spring Boot 3.2.4 + Java 17 |
| ORM | MyBatis-Plus 3.5.6 |
| 数据库 | MySQL 8.0+ |
| 缓存 | Redis (Lettuce) + Redisson 分布式锁 |
| 消息队列 | RabbitMQ |
| 安全 | Spring Security + JWT |
| 实时推送 | WebSocket（通知） + SSE（AI生成流式输出） |
| AI API | DeepSeek |
| 前端 | Vue 3 + TypeScript + Element Plus + Pinia |
| 容器化 | Docker + Docker Compose |


## 二、启动与部署

### Q: 本地开发怎么启动？
A: 需要启动 MySQL、Redis、RabbitMQ 三个中间件，然后运行后端。

**第一步：启动中间件（三种方式）**

方式一（Docker Compose，推荐）：
```bash
docker compose up -d mysql redis rabbitmq
```

方式二（Windows 本地装）：
- MySQL：用 phpStudy / XAMPP 或独立安装
- Redis：Windows 版 Redis 或 WSL 中运行
- RabbitMQ：下载 Windows 安装包

方式三（只启动 MySQL，其他用 Docker）：
```bash
docker run -d --name mysql8 -p 3306:3306 -e MYSQL_ROOT_PASSWORD=123456 mysql:8.0
```

**第二步：创建数据库**
```bash
mysql -u root -p < backend/sql/init.sql
```

**第三步：启动后端**
```bash
cd backend
mvn spring-boot:run
```

**第四步：启动前端**
```bash
cd ai-creator/frontend
npm install
npm run dev
```

访问 http://localhost:5173，默认管理员 admin/admin123

### Q: 生产部署怎么启动？

```bash
# 设置你的 DeepSeek API Key
export DEEPSEEK_API_KEY=sk-your-key-here

# 一键启动全部服务
docker compose up --build -d

# 访问 http://localhost
```

### Q: 数据库需要手动建吗？
A: 两种方式：
1. **首次部署**：执行 `mysql -u root -p < backend/sql/init.sql`
2. **Docker 部署**：容器首次启动会自动执行 `init.sql`

## 三、核心业务流程

### Q: AI 内容生成的完整流程是什么？

```
用户输入关键词 → 点击"开始生成"
        ↓
前端创建 EventSource 连接
   GET /api/ai/generate/sse?keywords=xxx&templateType=xxx&token=xxx
        ↓
后端 JwtAuthenticationFilter 从 query param 提取 token → 验证
        ↓
AiController 创建 SseEmitter（超时 120s）→ 调用 AiService.generateStream()
        ↓
AiService：
  1. 根据 templateType 查 ai_template 表 → 获取 system_prompt
  2. 构建 messages = [{role:system, content:system_prompt}, {role:user, content:用户输入}]
  3. 调用 DeepSeekClient.streamChat(messages, ...)
        ↓
DeepSeekClient：
  1. POST https://api.deepseek.com/v1/chat/completions （stream=true）
  2. BufferedReader 逐行读取响应
  3. 解析 data: {"choices":[{"delta":{"content":"..."}}]}
  4. 提取 content → 回调 onChunk
        ↓
AiService.onChunk → emitter.send(SseEmitter.event().name("message").data(json))
        ↓
前端 EventSource 收到 message 事件 → 追加到 generatedContent（打字机效果）
        ↓
生成完毕 → emitter.send(event().name("done")) → 前端显示"生成完成"
        ↓
用户点击"保存" → POST /api/contents → ContentService.save()
```

### Q: SSE 流式输出为什么不用 Axios？
A: `EventSource` 是浏览器原生 API，专门用于 SSE（Server-Sent Events）。它自动处理重连，适合服务端推流的场景。而 Axios 基于 XHR，对流式响应支持不完善。

但 `EventSource` 有个限制：**不能设置自定义请求头**。所以 JWT token 只能通过 URL 查询参数传递：
```
/api/ai/generate/sse?keywords=...&token=xxx
```
`JwtAuthenticationFilter` 中做了特殊处理：如果请求路径包含 `/api/ai/generate/sse` 且没有 Bearer header，就从 `request.getParameter("token")` 取。

### Q: 内容版本控制怎么实现的？

```
首次保存 → ContentService.create() → version = 1 存入 ai_content_version
二次编辑保存 → ContentService.update() → 查最大版本号 → version = 2 → 存入新快照
回滚 → ContentService.rollback(contentId, versionNumber)
  1. 查 ai_content_version 中指定版本号的记录
  2. 将 title/summary/content 覆盖到 ai_content 当前记录
  3. 记录一个新版本（version + 1）作为"回滚快照"
```

### Q: 通知系统怎么工作的？

```
某操作触发通知（如任务分配、积分变更）
        ↓
NotificationService.sendNotification()（@Async 异步执行）
  1. 通知数据 INSERT 到 sys_notification 表
  2. RabbitMQSender 将通知发布到 DirectExchange
        ↓
RabbitMQ 将消息路由到 notification 队列
        ↓
NotificationRabbitListener 消费消息
  1. 反序列化 Notification
  2. 序列化为 JSON
  3. 通过 NotificationWebSocketHandler.sendToUser() 推送给指定用户
        ↓
用户浏览器 WebSocket 收到通知 → 实时展示
```

## 四、常见面试问题

### Q: 为什么用 Redisson 而不是直接 RedisTemplate + SETNX？
A: Redisson 的 RLock 有"看门狗"机制 —— 获取锁后每 10 秒检查一次，如果任务没完成就自动续期到 30 秒。手写 SETNX 做不到这个，锁过期了都不知道。

### Q: RabbitMQ 在这里的作用是什么？
A: 解耦。通知发送方不需要知道谁来消费通知，只需要发到交换机。RabbitMQ 保证消息不丢失（队列持久化），即使 WebSocket 推送失败，消息也不会丢。

### Q: 为什么 Nginx 要配置 proxy_buffering off？
A: SSE 是流式响应，需要数据实时到达前端。Nginx 默认会缓冲代理响应（攒够一定量才发送），这会让流式输出变成"一段一段"的。关掉缓冲后数据随到随发。

### Q: 为什么前端 auth store 里 login 和 register 的逻辑不同？
A: login 成功后直接设置 token 并跳转到首页（用户已登录可以直接使用）。register 成功后只提示"注册成功，请登录"（需要用户再输一次密码登录）。这是安全设计——注册成功后不自动登录。

## 五、常见问题排查

### Q: 启动报 `Failed to configure DataSource`？
A: MySQL 没启动，或者 `application.yml` 里的连接信息不对。先确认 MySQL 在运行：

```bash
docker ps | grep mysql
```

### Q: SSE 连接上但收不到数据？
A: 检查两步：
1. 后端日志有没有 DeepSeek API 的报错
2. 如果用了 Nginx 代理，确保 `proxy_buffering off;` 已配置

### Q: Docker Compose 启动后前端访问后接口 502？
A: 后端启动比前端慢。等后端完全启动（看日志 `Started SmartTaskApplication`），或者加 `depends_on` + `healthcheck`。

### Q: `docker compose up` 报端口被占用？
A: 本地的 MySQL/Redis/RabbitMQ 占用了 3306/6379/5672 端口。要么关掉本地服务，要么改 `docker-compose.yml` 的端口映射。

## 六、内容 CRUD API 详解

### Q: ContentController 提供了哪些接口？

| 方法 | 路径 | 功能 | 请求体/参数 |
|------|------|------|------------|
| GET | `/api/contents?page=1&size=10` | 分页查询内容列表 | page/size/keyword/templateType |
| POST | `/api/contents` | 创建内容（含幂等保护） | ContentDTO JSON |
| GET | `/api/contents/{id}` | 获取内容详情 | — |
| PUT | `/api/contents/{id}` | 更新内容 | ContentDTO JSON |
| DELETE | `/api/contents/{id}` | 删除内容（逻辑删除） | — |
| POST | `/api/contents/{id}/favorite` | 切换收藏状态 | — |
| GET | `/api/contents/categories` | 获取分类列表 | — |
| POST | `/api/contents/categories` | 创建分类 | {"name": "分类名"} |
| DELETE | `/api/contents/categories/{id}` | 删除分类 | — |
| GET | `/api/contents/{id}/versions` | 获取版本历史 | — |
| POST | `/api/contents/{id}/rollback/{version}` | 回滚到指定版本 | — |

### Q: ContentService 的核心方法执行流程？

**createContent:**
```
ContentDTO → 构建 Content → INSERT → 保存 Version(version=1) → 返回 Content
```

**updateContent:**
```
查 Content → 校验创建者身份 → 保存当前快照(version+1) → 更新字段 → UPDATE
```

**rollback:**
```
查 Content → 校验创建者 → 查目标版本 → 保存当前快照 → 覆盖内容 → UPDATE → 保存回滚快照
```

### Q: @Idempotent 注解是干什么的？
A: 防止用户快速双击保存按钮导致重复提交。

用法：在 createContent 方法上加了 `@Idempotent(message = "内容正在保存中，请勿重复提交")`。

原理（在 IdempotentAspect.java 中实现）：
1. 从请求头取 `idempotent-token`
2. 用 Redisson 分布式锁保证原子性
3. 第一次请求：将 token 存入 Redis（过期时间 10 秒），执行原方法
4. 重复请求：发现 token 已存在，直接返回"请勿重复提交"

### Q: 为什么用 @RequestParam 传分页参数？
A: 前端 ContentList.vue 通过 GET 请求传递查询参数：
```
GET /api/contents?page=1&size=10&keyword=Java&templateType=article
```

`@RequestParam(defaultValue = "1") int page` 从 URL 查询字符串中取 page 的值，
defaultValue = "1" 表示前端没传 page 时默认值为 1。

### Q: 版本号怎么管理的？
A: 每次保存内容时调用 `saveVersionSnapshot()`：
- 首次保存 → version = 1
- 每次更新 → 先查当前最大版本号，然后 +1
- 回滚时 → 先保存当前状态为新版本，再覆盖内容，再保存回滚后状态为新版本

每次 save 都会创建一个 ContentVersion 记录，存在 `ai_content_version` 表里。

### Q: 为什么删除内容是"逻辑删除"？
A: Content 实体类中 `deleted` 字段加了 `@TableLogic` 注解。调用 `deleteById()` 时不是真的 DELETE，而是 UPDATE deleted = 1。查询时 MyBatis-Plus 自动拼接 `WHERE deleted = 0`。

好处：数据不丢失，可以恢复，也能保留关联的版本历史。
