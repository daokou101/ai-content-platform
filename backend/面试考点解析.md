## 1. Spring Security + JWT 认证流程

### Q：描述一下整个登录认证的过程

---

#### 正式版

登录时，前端传入 `{username, password}` 到 `/api/auth/login`。`AuthService.login()` 调用 `AuthenticationManager.authenticate()`，它会自动委托给 `CustomUserDetailsService.loadUserByUsername()` 从数据库查用户，然后用 `PasswordEncoder`（BCrypt）比对密码。认证成功后，生成 JWT Token，同时存入 Redis 用于支持强制下线。

之后每次请求，前端在 Header 带 `Authorization: Bearer <token>`。`JwtAuthenticationFilter`（继承 `OncePerRequestFilter`）从 Header 提取 Token，解析出用户信息，构建 `UsernamePasswordAuthenticationToken` 塞进 `SecurityContextHolder`，后续 Controller 通过 `@AuthenticationPrincipal` 获取当前用户。

---

#### 详细版

咱们一步步拆解登录的完整过程：

**第一步：用户发起登录请求**

用户在浏览器输入用户名和密码，点登录按钮。前端发一个 POST 请求：
```
POST /api/auth/login
Content-Type: application/json

{
  "username": "admin",
  "password": "123456"
}
```

**第二步：Controller 接收请求**

`AuthController.login()` 方法被调用，`@Valid` 注解先检查参数格式是否符合要求（用户名不能为空、密码长度等）。

然后调用 `authService.login(loginDTO)`。

**第三步：AuthenticationManager 登场**

```java
Authentication authentication = authenticationManager.authenticate(
    new UsernamePasswordAuthenticationToken(loginDTO.getUsername(), loginDTO.getPassword())
);
```

这里做了三件事：
1. `UsernamePasswordAuthenticationToken` 只是一个"凭证对象"，拿着用户名和密码
2. `authenticationManager.authenticate()` 是 Spring Security 的认证入口
3. 它会自动查找项目中的 `UserDetailsService` 实现——也就是我们的 `CustomUserDetailsService`

**第四步：加载用户信息**

`CustomUserDetailsService.loadUserByUsername("admin")` 被调用：
```java
User user = userMapper.selectOne(
    new LambdaQueryWrapper<User>().eq(User::getUsername, username)
);
```

从 `sys_user` 表中查到用户记录，如果找不到抛 `UsernameNotFoundException`。

查到后包装成 `CustomUserDetails` 返回，这个对象里有：用户密码（加密后的）、角色、状态等。

**第五步：密码比对**

Spring Security 拿到 `CustomUserDetails` 后，用 `PasswordEncoder`（BCrypt）比对：
- 用户输入的明文密码 → BCrypt 加密
- 和数据库存的加密密码对比
- 不匹配抛 `BadCredentialsException`

**第六步：生成 Token**

认证成功后，`authentication.getPrincipal()` 拿到 `CustomUserDetails`，生成 JWT：
```java
String token = jwtUtils.generateToken(user.getId(), user.getUsername(), user.getRole());
```

JWT 里包含了：
```json
{
  "sub": "1",         // 用户ID
  "username": "admin",
  "role": "ADMIN",
  "iat": 1716900000,  // 签发时间
  "exp": 1716986400   // 过期时间（24小时后）
}
```

**第七步：存 Redis**

把 Token 存到 Redis，key 为 `token:1`（1 是用户ID），24 小时过期：
```java
redisTemplate.opsForValue().set("token:1", token, 24, TimeUnit.HOURS);
redisTemplate.opsForSet().add("online:users", "1");  // 记录在线
```

**第八步：后续请求的认证**

用户之后的每一个请求，前端都会在 Header 带 JWT：
```
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiIxIiwidXNlcm5hbWUiOiJhZG1pbiJ9.xxx
```

`JwtAuthenticationFilter.doFilterInternal()` 被执行：
```java
// 1. 从请求头提取 Token
String token = extractToken(request);  // 去掉 "Bearer " 前缀

// 2. 验证 Token 签名和有效期
jwtUtils.validateToken(token)

// 3. 解析出用户名
String username = jwtUtils.getUsernameFromToken(token);

// 4. 重新加载用户信息
CustomUserDetails userDetails = loadUserByUsername(username);

// 5. 构造成 Authentication 对象
UsernamePasswordAuthenticationToken authentication =
    new UsernamePasswordAuthenticationToken(userDetails, null, userDetails.getAuthorities());

// 6. 放入 SecurityContext（Spring Security 的上下文）
SecurityContextHolder.getContext().setAuthentication(authentication);
```

这样后续的 Controller 方法就能通过 `@AuthenticationPrincipal CustomUserDetails userDetails` 拿到当前用户信息。

## 2. SecurityConfig 配置详解

### Q：SecurityConfig 里做了哪些配置？

---

#### 正式版

核心配置有三块：**禁用 CSRF**（JWT 天然免疫 CSRF）、**无状态会话**（不用 Session）、**自定义 JWT 过滤器**。白名单放行登录/注册/WebSocket/Swagger 路径，其余请求全部需要认证。同时声明了 `AuthenticationManager`（认证管理器）和 `PasswordEncoder`（BCrypt 密码编码器）的 Bean。

---

#### 详细版

`SecurityConfig` 是整个项目安全体系的"总开关"，逐行解释：

```java
@Configuration
@EnableWebSecurity                          // 开启 Spring Security 的 Web 安全功能
@EnableMethodSecurity                       // 开启 @PreAuthorize 等方法级注解
@RequiredArgsConstructor
public class SecurityConfig {
```

**1. 白名单数组**

```java
private static final String[] WHITE_LIST = {
    "/api/auth/login",          // 登录——没登录当然要能访问
    "/api/auth/register",       // 注册——同上
    "/api/auth/forgot-password",// 忘记密码
    "/ws/**",                   // WebSocket 连接（在拦截器里单独鉴权）
    "/v3/api-docs/**",          // Swagger 接口文档
    "/swagger-ui/**"
};
```

这些路径不需要登录也能访问。

**2. SecurityFilterChain（核心）**

```java
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http
        .csrf(csrf -> csrf.disable())
        // ^ 禁用 CSRF。CSRF 攻击靠的是浏览器自动带 Cookie，
        //   我们用的是 JWT（存在请求头里，不会自动带），所以 CSRF 不适用

        .sessionManagement(session ->
            session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
        // ^ 告诉 Spring Security：不要创建 Session，不要用 Session 存用户信息
        //   每一次请求都是独立的，全凭 JWT 认证

        .authorizeHttpRequests(auth -> auth
            .requestMatchers(WHITE_LIST).permitAll()
            .anyRequest().authenticated()
        )
        // ^ 白名单放行，其余所有请求必须登录

        .addFilterBefore(jwtAuthenticationFilter,
                         UsernamePasswordAuthenticationFilter.class);
        // ^ 把我们的 JWT 过滤器插入到过滤器链中
        //   位置是在 UsernamePasswordAuthenticationFilter 之前
        //   因为我们要先解析 JWT，再走后面的鉴权流程
    return http.build();
}
```

**3. 两个重要 Bean**

```java
@Bean
public AuthenticationManager authenticationManager(
        AuthenticationConfiguration authConfig) throws Exception {
    return authConfig.getAuthenticationManager();
}
// 登录时需要用这个来认证

@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}
// 密码加密和比对用的
```

**这里有个关键问题：为什么 SecurityFilterChain 用 @Bean 而不用继承？**

这是 Spring Security 3.x（对应 Spring Boot 3.x）的新写法。以前要继承 `WebSecurityConfigurerAdapter` 然后重写 `configure()` 方法。现在用 @Bean 返回 `SecurityFilterChain` 更灵活，更符合组合优于继承的原则。

## 3. JwtAuthenticationFilter 工作原理

### Q：JWT 过滤器是怎么工作的？为什么继承 OncePerRequestFilter？

---

#### 正式版

`OncePerRequestFilter` 确保每个请求只经过一次过滤。核心逻辑在 `doFilterInternal()`：从 Header 取 Token → 验证有效性 → 解析用户 → 塞入 `SecurityContextHolder`。这样做之后，项目中的任何地方都可以通过 `SecurityContextHolder.getContext().getAuthentication()` 获取当前用户，Spring Security 也知道这个请求是谁发的了。

---

#### 详细版

先解释 `OncePerRequestFilter`：

普通的 Filter 在"转发（forward）"场景下会被调用多次。比如请求 A 到 Servlet，Servlet 转发给另一个 Servlet，Filter 会执行两遍。`OncePerRequestFilter` 保证一次请求只执行一次 doFilter，不会重复。

**核心方法 `doFilterInternal` 逐行拆解：**

```java
@Override
protected void doFilterInternal(HttpServletRequest request,
                                HttpServletResponse response,
                                FilterChain filterChain)
                                throws ServletException, IOException {
```

**第 1 步：从请求头提取 Token**
```java
String token = extractToken(request);
// extractToken 方法：
//   1. 从 Header 取 "Authorization" 的值
//   2. 检查是不是以 "Bearer " 开头
//   3. 是的话去掉 "Bearer " 前缀，返回纯 Token
//   4. 不是的话返回 null
```

**第 2 步：验证并解析 Token**
```java
if (StringUtils.hasText(token) && jwtUtils.validateToken(token)) {
    // validateToken 内部会：
    //   1. 用密钥解密签名
    //   2. 检查是否过期
    //   3. 签名不对或过期了都会抛异常，catch 住返回 false
```

**第 3 步：从 Token 中获取用户信息**
```java
    String username = jwtUtils.getUsernameFromToken(token);
    CustomUserDetails userDetails =
        (CustomUserDetails) userDetailsService.loadUserByUsername(username);
    // 这里又查了一次数据库！每次请求都会查一次用户信息
    // 面试优化点：可以把用户信息也缓存到 Redis，减少数据库查询
```

**第 4 步：构建认证对象，放入 SecurityContext**
```java
    UsernamePasswordAuthenticationToken authentication =
        new UsernamePasswordAuthenticationToken(
            userDetails, null, userDetails.getAuthorities()
        );
    // 三个参数版本 = 已认证状态（authenticated = true）
    // 两个参数版本 = 未认证状态

    SecurityContextHolder.getContext().setAuthentication(authentication);
    // SecurityContextHolder 底层是 ThreadLocal
    // 每个请求的线程独立，互不干扰
    // 请求结束后 SecurityContextHolderFilter 会自动清除，防止内存泄漏
}
```

**第 5 步：放行**
```java
    filterChain.doFilter(request, response);
    // 不管有没有认证成功，都要放行
    // 没认证的请求会在后面被 Security 拦截
}
```

## 4. BCrypt 密码加密

### Q：为什么用 BCrypt 而不用 MD5？

---

#### 正式版

MD5 是哈希算法而非密码加密算法，计算速度极快，用彩虹表可以反向查原文。BCrypt 设计时专门针对密码场景做了三件事：**内置盐值（salt）** 使相同密码每次加密结果不同；**可调 cost 参数**控制计算速度（cost=10 时单次加密约 100ms，暴力破解成本极高）；**盐值自动存储在密文中**，验证时无需额外存储。

---

#### 详细版

**MD5 的问题：**

假设我用 MD5 加密密码 `123456` 得到 `e10adc3949ba59abbe56e057f20f883e`。攻击者可以：
1. 预先计算常见密码的 MD5 值，做成彩虹表
2. 看到你的密文 `e10adc...`，反向查表就知道明文是 `123456`
3. MD5 计算极快，现代 GPU 每秒能算几十亿次

**BCrypt 做了三件事解决这个问题：**

**1. 内置盐值**

BCrypt 加密时会自动生成一个随机 salt，把 salt 和密码混在一起算：
```
第一次加密 "123456" → $2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy
第二次加密 "123456" → $2a$10$k2t4ZQ8jK1qJi5YgLx9Yxu3Zn3Q5O5jGZxL5X5b5d5e5f5g5h5i5j5k5
```

两次结果完全不一样！攻击者没法用彩虹表匹配。

而且这个密文里其实包含了 salt：
```
$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy
  ││  │           │
  ││  │           └── 哈希值
  ││  └────────────── 盐值（存储在这里！）
  │└───────────────── cost 参数（2^10 = 1024 轮加密）
  └────────────────── BCrypt 版本标识
```

验证时 BCrypt 自动从密文中取出盐值，用同样的盐值加密输入的明文，看结果是否一致。

**2. cost 参数可以调**

```java
new BCryptPasswordEncoder()      // 默认 cost=10，约 1024 轮
new BCryptPasswordEncoder(12)    // 约 4096 轮，更安全但也更慢
```

cost=10 时单次验证约 100ms，用户感觉不到。但对攻击者来说，尝试 100 万个常见密码需要 `100ms × 100万 = 100,000 秒 ≈ 28 小时`，成本高得多。

**3. BCrypt 不可逆**

MD5 虽然理论上不可逆，但彩虹表让它在实践中"可逆"。BCrypt 因为每次结果不同 + 计算慢，彩虹表几乎不可行。

## 5. Redisson 分布式锁 & 幂等性

> 这是项目中**面试价值最高的模块**，对应文件：`IdempotentAspect.java`、`RedissonConfig.java`、`PointsService.java`

### Q：描述一下 @Idempotent 幂等注解的实现原理

---

#### 正式版

`@Idempotent` 注解通过 AOP 切面拦截带有该注解的方法。请求到达时，切面从请求头中获取 `idempotent-token`，先尝试获取 Redisson 分布式锁（`tryLock(0, 5, SECONDS)`，0 表示拿不到立即返回），获取锁后检查 Redis 中是否存在该 token 的处理记录。如果是第一次请求，存入 Redis 并设置过期时间，然后执行原方法；如果是重复请求（Redis 中已存在），直接返回"请勿重复提交"。最后在 finally 块中释放锁。

锁的作用是防止并发下两个请求同时通过检查，Redis 标记的作用是请求去重。

---

#### 详细版

**先理解业务场景：**

用户在浏览器点击"创建任务"，网络卡了一下，网页没响应，用户又点了一下。

没有幂等的情况下：后端收到了两次完全相同的请求 → 创建了两个一模一样的任务 → 用户一头雾水。

加上幂等性后：第一个请求正常创建，第二个请求直接被拦截返回"请勿重复提交"。

**现在看代码实现：**

我们自己在方法上加 `@Idempotent` 注解：
```java
@PostMapping
@Idempotent(message = "任务正在创建中，请勿重复提交")
public Result<Task> createTask(@Valid @RequestBody TaskDTO taskDTO, ...) {
    return Result.success(taskService.createTask(taskDTO, ...));
}
```

**切面 `IdempotentAspect` 的完整执行流程：**

```java
@Around("@annotation(idempotent)")
public Object around(ProceedingJoinPoint joinPoint, Idempotent idempotent) throws Throwable {
```

**第一步：取幂等键**

前端在创建任务时，请求头里带一个唯一标识：
```
idempotent-token: uuid-xxxx-xxxx
```
这个 UUID 由前端生成，每次新操作生成一个新的 UUID。

切面从请求头取出这个值：
```java
String idempotentKey = request.getHeader("idempotent-token");
if (idempotentKey == null || idempotentKey.isEmpty()) {
    return joinPoint.proceed();  // 没传就放行
}
```

**第二步：获取分布式锁**

```java
RLock lock = redissonClient.getLock("idempotent:lock:" + idempotentKey);
boolean locked = lock.tryLock(0, 5, TimeUnit.SECONDS);
```

这里用 `tryLock(0, 5, SECONDS)` 的详细含义：
- 第一个参数 `0` = **等待时间**（waitTime）
  - 拿不到锁时不等待，立即返回 false
  - 为什么是 0？因为两个完全相同的请求几乎同时到达的概率极低，不值得等待
  - 而且我们的场景：如果拿不到锁，说明已经有请求在处理了，我应该直接返回"重复请求"而不是排队等着
- 第二个参数 `5` = **租约时间**（leaseTime）
  - 获取锁之后 5 秒自动释放
  - 防止拿到锁的线程挂了导致死锁
  - 注意：设了 leaseTime 后 Redisson 的 Watchdog**不会启动**，因为不需要续期

**第三步：检查 Redis 中是否已有处理记录**

```java
boolean alreadyProcessed = Boolean.TRUE.equals(
    redissonClient.getBucket("idempotent:key:" + idempotentKey).isExists()
);
```

- 第一次请求：Redis 中没有这个 key → `alreadyProcessed = false`
- 重复请求：Redis 中有这个 key → `alreadyProcessed = true` → 直接拒绝

**第四步：标记处理 + 执行原方法**

```java
// 标记为已处理，10 秒后自动过期
redissonClient.getBucket("idempotent:key:" + idempotentKey)
    .set(true, idempotent.expireTime(), TimeUnit.SECONDS);

// 执行原 Controller 方法
return joinPoint.proceed();
```

**第五步：释放锁**

```java
finally {
    lock.unlock();
}
```

**为什么锁和 Redis 标记两个都要？**

用一个例子说明：

| 时间 | 请求 A | 请求 B |
|------|--------|--------|
| T1 | 获取锁 ✅ | - |
| T2 | 检查 Redis：不存在 ✅ | - |
| T3 | 标记 Redis 为已处理 | - |
| T4 | 释放锁 | - |
| T5 | 执行业务方法（创建任务） | - |
| T6 | - | 获取锁 ✅ |
| T7 | - | 检查 Redis：已存在 ❌ 拒绝 |

**如果只有锁，没有 Redis 标记：**
| 时间 | 请求 A | 请求 B |
|------|--------|--------|
| T1 | 获取锁 ✅ | - |
| T2 | 执行业务方法 | - |
| T3 | 释放锁 | - |
| T4 | - | 获取锁 ✅ |
| T5 | - | 检查？没有标记可查 → 又执行了一遍 ❌ |

**如果只有 Redis 标记，没有锁：**
| 时间 | 请求 A | 请求 B |
|------|--------|--------|
| T1 | 检查 Redis：不存在 ✅ | 检查 Redis：不存在 ✅（同时到达） |
| T2 | 标记 Redis | 标记 Redis ✅（覆盖了） |
| T3 | 执行业务方法 | 执行业务方法 ❌ 重复了 |

**所以：锁解决并发冲突，Redis 标记做请求去重，缺一不可。**

---

### Q：什么是 Redisson 的 Watchdog 机制？

---

#### 正式版

Watchdog 是 Redisson 的自动续期机制。当获取锁时不指定 leaseTime（默认 -1），Redisson 会给锁设一个 30 秒的默认过期时间，同时启动一个后台定时任务，每 10 秒检查一次：如果锁还在被持有，就把过期时间重置为 30 秒。这个机制防止了业务执行时间超过锁的过期时间，导致锁提前释放引发并发问题。如果持有锁的服务挂了，Watchdog 也会停掉，锁最多 30 秒自动释放。

---

#### 详细版

**Redis SETNX 手写锁的问题：**

假设我用 Redis SETNX 实现锁：
```java
// 获取锁：key = "myLock"，value = "thread-1"，过期 10 秒
SET myLock thread-1 NX PX 10000
```

问题来了：如果业务执行需要 15 秒，第 10 秒锁自动过期了，其他线程就能获取到锁。而线程 1 还在执行，到第 15 秒执行完一 unlock，把线程 2 的锁给解了。

**Redisson 怎么解决的？**

关键代码：`lock.tryLock(-1, -1, TimeUnit.SECONDS)` 或直接 `lock.lock()`（不传 leaseTime）

内部流程：
```
lock.lock()
  │
  ├── 1. Redis SET key value NX PX 30000（30秒过期）
  │
  ├── 2. 启动 Watchdog（一个后台定时任务）
  │     └── 每 10 秒执行一次：
  │           ├── 检查 Redis 中锁是否还在
  │           │   ├── 还在 → PEXPIRE key 30000（续期到 30 秒）
  │           │   └── 不在了 → 停止 Watchdog
  │           └── 检查当前线程是否还持有锁（可重入计数）
  │
  ├── 3. 执行业务逻辑（需要多久都行，Watchdog 会自动续期）
  │
  └── 4. unlock() 释放锁，关闭 Watchdog
```

**Watchdog 的核心价值：**

1. **防提前过期**：业务执行 50 秒，Watchdog 会自动续期，锁一直有效
2. **防死锁**：如果服务挂了，Watchdog 线程也挂了，没人续期，锁最多 30 秒后自动释放
3. **不需要预估算执行时间**：用 SETNX 你要猜业务要多久，猜短了提前过期，猜长了死锁时间也长

**项目中 @Idempotent 为什么没用到 Watchdog？**

因为前面 `tryLock(0, 5, SECONDS)` 传了 leaseTime = 5，Redisson 认为你已经明确指定了租约时间，所以 Watchdog 不启动。幂等性场景下业务执行通常很快（几十毫秒），5 秒完全够用，不需要 Watchdog。

## 7. Redis 缓存三大问题

### Q：什么是缓存穿透、缓存击穿、缓存雪崩？怎么解决？

---

#### 正式版

这三个是面试必问的体系化问题：

**缓存穿透**：查询不存在的数据，缓存没有，数据库也没有，每次请求都打到数据库。解决：缓存空值（即使数据库返回 null 也缓存）或布隆过滤器（快速判断 key 是否可能存在）。

**缓存击穿**：一个热点 key 过期瞬间，大量并发请求同时涌入数据库。解决：分布式锁 + 双重检查（第一个请求拿锁查数据库回写缓存，后续请求从缓存取），或者热点 key 设置永不过期。

**缓存雪崩**：大量 key 同一时间过期或 Redis 挂了。解决：过期时间加随机值避免同时过期；Redis 高可用（主从+哨兵/集群）；多级缓存（本地缓存 + Redis）。

---

#### 详细版

**问题一：缓存穿透**

场景：一个攻击者不断请求 ID = -1 的用户详情。
```
请求 -1 → 查缓存 → 没有 → 查数据库 → 也没有 → 返回 null
请求 -1 → 查缓存 → 没有 → 查数据库 → 也没有 → 返回 null
请求 -1 → 查缓存 → 没有 → 查数据库 → 也没有 → 返回 null
... 反复刷，数据库压力巨大
```

缓存穿透的核心问题：**缓存不存在的 key 没有意义**（因为 null 不会被缓存）。

解决方案有两种：

方案 1：缓存空值
```java
User user = redisTemplate.opsForValue().get("user:" + id);
if (user != null) {
    return user;
}
user = userMapper.selectById(id);
// 不管找没找到，都写缓存
redisTemplate.opsForValue().set("user:" + id, user, 5, TimeUnit.MINUTES);
// 如果 user 是 null，缓存的也是 null，下次直接从缓存返回
return user;
```

方案 2：布隆过滤器（Bloom Filter）
- 提前把所有用户 ID 存到一个超大的 bitmap 中
- 请求来的时候先检查 ID 在不在 bitmap 里
- 不在 → 直接返回，不查数据库
- 在 → 可能误判（布隆过滤器说"在"不一定在，说"不在"一定不在）
- 适合数据量极大的场景

**问题二：缓存击穿**

场景：某个热点数据（比如某个超级大 V 的首页数据），缓存刚好过期。一瞬间来了 10000 个请求。

没有防护：
```
请求 1 → 缓存过期 → 查数据库 → 回写缓存
请求 2 → 缓存过期 → 查数据库 → 回写缓存
请求 3 → 缓存过期 → 查数据库 → 回写缓存
... 10000 个请求全打到数据库
```

用分布式锁 + 双重检查：
```
请求 1 → 缓存过期 → 获取锁 ✅ → 再查缓存（双重检查）→ 还是没有 → 查数据库 → 回写缓存
请求 2 → 缓存过期 → 获取锁 ❌（等待）→ 等到了 → 查缓存 ✅ 有数据了 → 直接返回
请求 3 → 缓存过期 → 获取锁 ❌（等待）→ 等到了 → 查缓存 ✅ 有数据了 → 直接返回
...
```

**问题三：缓存雪崩**

场景一：双十一 0 点，1000 个商品设置了 0 点过期，过期后 100 万用户刷新，1000 个请求同时打到数据库。

场景二：Redis 服务宕机，所有请求直接到数据库。

解决方案：

场景一（大量 key 同时过期）：
```java
// 不要设固定过期时间，加随机值
int baseExpire = 3600;  // 基础 1 小时
int random = new Random().nextInt(600);  // 随机 0-10 分钟
redisTemplate.opsForValue().set(key, value, baseExpire + random, TimeUnit.SECONDS);
```

场景二（Redis 宕机）：
- 主从 + 哨兵：主节点挂了哨兵自动选一个从节点升主
- Redis Cluster：数据分片存储，部分节点挂了不影响整体
- 本地缓存（Caffeine/Guava）：Redis 挂了，本地缓存还能扛一下

## 8. RabbitMQ 消息队列

### Q：描述项目中 RabbitMQ 的使用场景和架构

---

#### 正式版

项目使用 RabbitMQ 解耦通知发送和通知推送。当任务状态变更或触发事件时，业务代码调用 `RabbitMQSender.sendNotification()` 向 `smart.task.direct` 直连交换机发送消息，路由键为 `notification`。消息被路由到 `smart.task.notification` 队列（持久化），`NotificationRabbitListener` 消费消息并通过 WebSocket 推送给目标用户。

引入 MQ 的好处有三层：**解耦**（发送方不需要知道消息如何推送）、**可靠性**（消息持久化，重启不丢）、**削峰**（MQ 作为缓冲区，消费者按能力消费）。

---

#### 详细版

**为什么需要消息队列？**

没有 MQ 的时候，如果某个服务需要发通知给用户，代码可能是这样的：
```java
// 业务代码直接调用 WebSocket 推送
webSocketHandler.sendToUser(userId, "你有新的通知");
```

问题：
1. 如果 WebSocket 服务挂了，通知就丢了
2. 如果其他服务也要发通知，都得知道 WebSocket 怎么用
3. 如果瞬间 1000 个通知，WebSocket 推送不过来，业务线程就被阻塞了

引入 MQ 后：

**第一步：配置交换机、队列和绑定**

```java
// RabbitMQConfig.java
// 创建直连交换机
@Bean
public DirectExchange directExchange() {
    return new DirectExchange("smart.task.direct");
}

// 创建通知队列（durable=true 表示重启后队列还在）
@Bean
public Queue notificationQueue() {
    return QueueBuilder.durable("smart.task.notification").build();
}

// 把队列绑定到交换机，路由键 = "notification"
@Bean
public Binding notificationBinding(DirectExchange exchange, Queue queue) {
    return BindingBuilder.bind(queue).to(exchange).with("notification");
}
```

这样 RabbitMQ 里就有了：
```
        smart.task.direct（交换机）
               │
               │ 路由键 = "notification"
               ▼
   smart.task.notification（队列，持久化）
```

**第二步：发送消息**

```java
// RabbitMQSender.java
public void sendNotification(Notification notification) {
    rabbitTemplate.convertAndSend(
        "smart.task.direct",     // 交换机
        "notification",          // 路由键
        notification              // 消息体（自动序列化为 JSON）
    );
}
```

业务代码调这个方法时，根本不需要知道通知最终是怎么推送给用户的——这就是解耦。

**第三步：消费消息**

```java
// NotificationRabbitListener.java
@RabbitListener(queues = "smart.task.notification")
public class NotificationRabbitListener {

    @RabbitHandler
    public void handleNotification(Notification notification) {
        // 从 MQ 拿到消息
        // 通过 WebSocket 推送给用户
        boolean sent = webSocketHandler.sendToUser(notification.getUserId(), jsonMessage);
        if (!sent) {
            // 用户不在线，通知已存数据库，上线后再展示
        }
    }
}
```

**消息的完整流转：**
```
任务状态变更
    │
    ▼
业务代码调用 sendNotification()
    │
    ▼
RabbitTemplate.convertAndSend()  →  交换机 smart.task.direct
                                       │
                                       │ 匹配路由键 "notification"
                                       ▼
                                  队列 smart.task.notification（持久化存储）
                                       │
                                       │ 消费者拉取
                                       ▼
                                  NotificationRabbitListener
                                       │
                                       ▼
                                  WebSocket → 浏览器
```

**第四步（面试可讲的扩展）：消息可靠性**

当前项目实现是基础版，面试时可以补充说明知道怎么改进：

**生产者确认**：
```yaml
spring.rabbitmq.publisher-confirm-type=correlated
```
加了之后，消息有没有成功到达交换机，会回调一个 ConfirmCallback 通知你。

**消费者手动 ACK**：
```yaml
spring.rabbitmq.listener.simple.acknowledge-mode=manual
```
当前是自动 ACK（消息一消费就确认），如果处理异常消息就丢了。改手动 ACK 后，处理失败的消息会重新入队。

**死信队列**：
```java
// 声明死信队列
@Bean
public Queue notificationDlq() {
    return QueueBuilder.durable("notification.dlq").build();
}
// 处理失败 N 次后 → 进入死信队列 → 人工排查
```

## 9. WebSocket 实时推送

### Q：WebSocket 是怎么和用户建立连接并推送通知的？

---

#### 正式版

客户端连接 `ws://host/ws/{userId}?token=xxx` 时，`WebSocketAuthInterceptor` 在握手阶段从 URL 参数取 JWT 进行鉴权，验证通过后将 userId 存入 session attributes。连接建立后，
`NotificationWebSocketHandler` 把 session 存入 `ConcurrentHashMap<userId, session>`。当需要推送通知时（MQ 消费者触发的），调用 `sendToUser(userId, message)` 从 Map 中找到对应 session 发送消息。
如果用户不在线，返回 false，通知保留在数据库中，用户登录后通过 HTTP 拉取未读通知。

---

#### 详细版

**第一步：客户端建立 WebSocket 连接**

前端 JavaScript：
```javascript
const ws = new WebSocket("ws://localhost:8080/ws/1?token=eyJhbGciOiJIUzI1Ni...");
```

URL 里带上了 token 用于鉴权，路径中的 `1` 是用户 ID。为什么不在连接后再鉴权？因为 WebSocket 握手阶段走的是 HTTP 协议，我们可以在握手时拦截。

**第二步：握手拦截器鉴权**

```java
// WebSocketAuthInterceptor.java
public boolean beforeHandshake(ServerHttpRequest request, ...) {
    if (request instanceof ServletServerHttpRequest servletRequest) {
        HttpServletRequest httpRequest = servletRequest.getServletRequest();
        String token = httpRequest.getParameter("token");
        if (token != null && jwtUtils.validateToken(token)) {
            Long userId = jwtUtils.getUserIdFromToken(token);
            attributes.put("userId", userId);  // 把 userId 存入 session
            return true;  // 允许连接
        }
    }
    return false;  // Token 无效，拒绝连接
}
```

**第三步：连接建立，记录会话**

```java
// NotificationWebSocketHandler.java
public void afterConnectionEstablished(WebSocketSession session) {
    Long userId = (Long) session.getAttributes().get("userId");
    sessions.put(userId, session);  // 存入 Map
    // sessions 是 ConcurrentHashMap<Long, WebSocketSession>
    log.info("用户 {} 已建立 WebSocket 连接", userId);
}
```

`ConcurrentHashMap` 是线程安全的 HashMap，因为 WebSocket 是多线程环境（可能有多个线程同时读写这个 Map）。

**第四步：推送通知**

当有通知需要发送时（比如任务被分配给了某个用户），MQ 消费者会触发：

```java
// NotificationWebSocketHandler.java
public boolean sendToUser(Long userId, String message) {
    WebSocketSession session = sessions.get(userId);
    if (session != null && session.isOpen()) {
        session.sendMessage(new TextMessage(message));
        return true;  // 推送成功
    }
    return false;  // 用户不在线
}
```

**第五步：连接断开，清理会话**

```java
public void afterConnectionClosed(WebSocketSession session, CloseStatus status) {
    Long userId = (Long) session.getAttributes().get("userId");
    sessions.remove(userId);  // 从 Map 中移除
}
```

**第六步：不在线怎么办？**

用户不在线时，通知已经存到数据库了（`sys_notification` 表）。用户下次登录时，前端调 HTTP 接口拉取未读通知：
```java
GET /api/notifications/unread-count  →  返回未读数
GET /api/notifications               →  返回通知列表
```

这就是**数据库 + WebSocket 双保险**。

**这个方案的一个局限性（面试高频追问）：**

`ConcurrentHashMap` 存在单台服务器的内存里，如果项目部署了 3 台服务器：
```
用户 A 连上了服务器 1 → sessions Map 在服务器 1
通知发到了服务器 2     → 在服务器 2 的 Map 里找用户 A → 找不到 → 返回不在线
```

解决方案可以讲：
```
方案 1：Redis 发布订阅
  - 通知到达任意一台服务器，都通过 Redis Pub/Sub 广播给所有服务器
  - 所有服务器都能找到自己的本地连接

方案 2：MQ 广播
  - 交换机类型改为 Fanout（扇形交换机），所有服务器都监听同一个队列

方案 3：WebSocket 代理
  - 用 Nginx 做 WebSocket 负载均衡（但存在 session 粘连问题）
  - 或集成 Socket.IO / Netty 的集群方案
```

## 10. 全局异常处理

### Q：`@RestControllerAdvice` 是怎么工作的？项目中处理了哪些异常？

---

#### 正式版

`@RestControllerAdvice` 是 `@ControllerAdvice + @ResponseBody` 的组合注解，本质是 AOP 中的"环绕增强"——它为所有 Controller 方法添加了异常拦截能力。项目中的 `GlobalExceptionHandler` 用 `@ExceptionHandler` 分类处理了四类异常：自定义业务异常（`BusinessException`，返回业务状态码）、参数校验异常（`MethodArgumentNotValidException` 等，返回 400）、权限不足异常（`AccessDeniedException`，返回 403）、以及兜底的所有未处理异常（返回 500 通用提示，防止系统异常信息泄露到前端）。

---

#### 详细版

**没有全局异常处理的时候：**

每个 Controller 方法都要写 try-catch：
```java
@PostMapping
public Result<Task> createTask(...) {
    try {
        // 业务逻辑
    } catch (BusinessException e) {
        return Result.failed(e.getCode(), e.getMessage());
    } catch (Exception e) {
        return Result.failed("系统错误：" + e.getMessage());
    }
}
```

10 个方法就要写 10 遍差不多的代码。而且容易遗漏，万一漏了，异常直接返回给前端，甚至暴露堆栈信息。

**全局异常处理怎么做：**

```java
@RestControllerAdvice
public class GlobalExceptionHandler {
```

加了这个注解，Spring 就知道这个类是用来拦截所有 Controller 的异常的。原理是 Spring AOP——在 Controller 方法执行时，如果抛了异常，Spring 会根据异常类型找对应的 `@ExceptionHandler` 方法。

**分类处理四种异常：**

```java
// 1. 自定义业务异常
@ExceptionHandler(BusinessException.class)
public Result<Void> handleBusinessException(BusinessException e) {
    // BusinessException 是我们自己定义的异常，包含 code 和 message
    // 比如：用户名已存在（4003）、积分不足（4004）
    return Result.failed(e.getCode(), e.getMessage());
}

// 2. 参数校验失败
@ExceptionHandler({MethodArgumentNotValidException.class, BindException.class, ConstraintViolationException.class})
public Result<Void> handleValidationException(Exception e) {
    // 当 @Valid 校验不通过时抛出的异常
    // 比如用户名是 @NotBlank 但前端传了空字符串
    return Result.failed(400, e.getMessage());
}

// 3. Spring Security 权限不足
@ExceptionHandler(AccessDeniedException.class)
public Result<Void> handleAccessDeniedException(AccessDeniedException e) {
    // 普通用户越权访问管理员接口时触发
    return Result.failed(403, "没有权限，请联系管理员");
}

// 4. 兜底——所有未处理异常
@ExceptionHandler(Exception.class)
public Result<Void> handleException(Exception e) {
    log.error("系统异常: ", e);  // 打印完整堆栈给后端排查
    return Result.failed("服务器繁忙，请稍后重试");  // 返回模糊信息给前端
}
```

**注意第四种"兜底"为什么重要：**

如果不兜底，未知异常会被 Tomcat 默认处理，可能返回 Tomcat 的错误页面（HTML）或者堆栈信息给前端（包含敏感信息、代码路径），这是安全漏洞。兜底后统一返回 JSON，message 是"服务器繁忙"这种安全提示，而详细的错误堆栈只打印到后端日志里。

## 11. 参数校验

### Q：`@Valid` 和 `@Validated` 有什么区别？怎么用？

---

#### 正式版

`@Valid` 是 Jakarta Validation 标准注解（JSR-380），支持嵌套校验。`@Validated` 是 Spring 对 `@Valid` 的增强，额外支持分组校验（不同场景使用不同的校验规则）。项目中 Controller 参数用 `@Valid` 触发 `LoginDTO` 或 `RegisterDTO` 中的 `@NotBlank`、`@Email` 等注解进行校验，校验不通过由 `GlobalExceptionHandler` 统一处理。

---

#### 详细版

**先看项目中怎么用的：**

```java
// AuthController.java
@PostMapping("/login")
public Result<LoginVO> login(@Valid @RequestBody LoginDTO loginDTO) {
    return authService.login(loginDTO);
}
```

```java
// LoginDTO.java
public class LoginDTO {
    @NotBlank(message = "用户名不能为空")
    private String username;
    @NotBlank(message = "密码不能为空")
    @Size(min = 6, max = 20, message = "密码长度6-20位")
    private String password;
}
```

当请求到达时，Spring 在反序列化 JSON 为 `LoginDTO` 对象后，会自动对加了校验注解的字段进行验证。

验证不通过时，**不会**执行 `authService.login()`，而是直接抛出 `MethodArgumentNotValidException`，被 `GlobalExceptionHandler` 捕获并返回错误信息。

**常用校验注解：**

| 注解 | 作用 | 示例 |
|------|------|------|
| `@NotBlank` | 字符串不能为 null 且去掉空格后不能为空 | 用户名、密码 |
| `@NotNull` | 值不能为 null | 必填数字 |
| `@Size(min, max)` | 字符串/集合长度范围 | 密码 6-20 位 |
| `@Email` | 邮箱格式 | 注册时邮箱 |
| `@Min / @Max` | 数值范围 | 年龄 |
| `@Pattern(regexp)` | 正则匹配 | 手机号 |

**`@Valid` vs `@Validated` 的区别：**

关键区别是**分组校验**。

假设有个 UserDTO，创建用户时 id 不用传（自增），更新用户时 id 必须传：
```java
public class UserDTO {
    private Long id;       // 创建时不用，更新时必填
    private String name;   // 创建和更新都需要
}
```

用 `@Validated` 分组可以做到：
```java
// 定义分组接口
public interface CreateGroup {}
public interface UpdateGroup {}

public class UserDTO {
    @Null(groups = CreateGroup.class)     // 创建时 id 必须为 null
    @NotNull(groups = UpdateGroup.class)  // 更新时 id 不能为 null
    private Long id;

    @NotBlank
    private String name;
}

// Controller 指定用哪个分组
@PostMapping
public Result create(@Validated(CreateGroup.class) @RequestBody UserDTO dto) {...}

@PutMapping
public Result update(@Validated(UpdateGroup.class) @RequestBody UserDTO dto) {...}
```

`@Valid` 不支持分组，所以如果项目需要分组校验要用 `@Validated`。

## 12. API 限流

### Q：项目中的限流是怎么实现的？有什么改进空间？

---

#### 正式版

项目使用固定窗口算法限流：`RateLimitInterceptor` 拦截所有 `/api/**` 请求（排除登录注册），对每个 IP 用 Redis 的 INCR 命令统计 10 秒内的请求次数，超过 10 次返回 429。这种方式实现简单，但存在窗口切换时的流量尖刺（窗口末尾 + 窗口开头短时间内可通过 2 倍阈值）。改进方向：滑动窗口（Redis ZSet 记录每次请求时间戳）或令牌桶算法。

---

#### 详细版

**限流代码逐行分析：**

```java
// RateLimitInterceptor.java
public boolean preHandle(HttpServletRequest request, ...) {
    String clientIp = request.getRemoteAddr();
    String redisKey = "rate:limit:" + clientIp;

    // INCR 命令：key 不存在时初始化为 0 然后 +1，存在时直接 +1
    // 这个操作是原子的（Redis 单线程）
    Long count = redisTemplate.opsForValue().increment(redisKey);

    if (count != null && count == 1) {
        // 第一个请求：设过期时间 10 秒
        redisTemplate.expire(redisKey, 10, TimeUnit.SECONDS);
    }

    if (count != null && count > 10) {
        // 超过阈值：返回 429 Too Many Requests
        response.setStatus(429);
        response.getWriter().write(...);
        return false;  // 不放行
    }

    return true;  // 放行
}
```

登录注册被排除在外（防止用户因为频繁登录被锁）。

**固定窗口算法的缺陷：**

假设 10 秒窗口允许 10 次请求：
```
第 1 秒窗口：请求 10 次（还剩 0 次额度）
第 10 秒窗口切换：清空计数器
第 11 秒（第 2 个窗口第 1 秒）：又请求 10 次

结果：第 9 秒到第 11 秒，2 秒内请求了 20 次！
```

这就是窗口切换时的流量尖刺问题。

**改进方案一：滑动窗口（用 ZSet）**

```java
String key = "sliding:limit:" + clientIp;
long now = System.currentTimeMillis();
long window = 10000;  // 10 秒

// 移除窗口外的记录
redisTemplate.opsForZSet().removeRangeByScore(key, 0, now - window);

// 获取当前窗口内请求数
Long count = redisTemplate.opsForZSet().zcard(key);

if (count < 10) {
    // 添加当前请求的时间戳
    redisTemplate.opsForZSet().add(key, String.valueOf(now), now);
    return true;  // 放行
}
return false;  // 限流
```
滑动窗口看的是当前时间往前 10 秒内有多少请求，没有窗口边界，流量更平滑。

**改进方案二：令牌桶算法**

原理：一个桶里最多 10 个令牌，每秒往桶里加 1 个令牌。请求来了拿令牌，拿到就处理，拿不到就等或拒绝。优势是可以应对突发流量（桶里最多攒 10 个令牌，能一口气处理 10 个请求），但平均速率是限制住的。

```java
// Guava RateLimiter（单机版）
RateLimiter rateLimiter = RateLimiter.create(10);  // 每秒 10 个
rateLimiter.tryAcquire();  // 拿到返回 true，拿不到返回 false
```
但 Guava 是单机的，多实例部署需要各实例分别计算。

## 13. AOP 操作日志

### Q：项目中的操作日志是怎么实现的？AOP 有哪些核心概念？

---

#### 正式版

使用 Spring AOP 的 `@Before` + `@AfterReturning` 在 Controller 层记录操作日志。`@Pointcut` 切点表达式 `execution(* com.smarttask.controller..*.*(..))` 匹配 controller 包和子包下的所有方法。前置通知打印请求信息（用户、IP、接口、时间），后置通知打印返回结果。这样所有 Controller 方法自动获得日志记录能力，无需在每个方法中重复写日志代码。

---

#### 详细版

**切面代码逐行解释：**

```java
@Aspect       // 告诉 Spring：这是一个切面类
@Component    // 让 Spring 管理这个 Bean
@Slf4j        // 日志注解
public class OperationLogAspect {

    // 定义切点
    @Pointcut("execution(* com.smarttask.controller..*.*(..))")
    public void controllerPointcut() {}
}
```

**切点表达式拆解：**
```
execution(* com.smarttask.controller..*.*(..))
          │         │               │  │  │
          │         │               │  │  └── 方法参数任意
          │         │               │  └───── 方法名任意
          │         │               └──────── 类名任意
          │         └───────────────────────── controller 包及子包
          └─────────────────────────────────── 返回值任意
```

**前置通知：**
```java
@Before("controllerPointcut()")
public void beforeController(JoinPoint joinPoint) {
    // 1. 获取当前请求的 Request 对象
    HttpServletRequest request = RequestContextUtils.getRequest();

    // 2. 从 SecurityContext 获取当前登录用户
    Authentication auth = SecurityContextHolder.getContext().getAuthentication();
    String username = "anonymous";
    if (auth != null && auth.getPrincipal() instanceof CustomUserDetails userDetails) {
        username = userDetails.getUsername();
    }

    // 3. 打印日志信息
    log.info("[操作日志] 用户: {} | IP: {}", username, RequestContextUtils.getClientIp());
    log.info("[操作日志] 接口: {} {}", request.getMethod(), request.getRequestURI());
    log.info("[操作日志] 方法: {}.{}", controller名, 方法名);
}
```

**后置通知：**
```java
@AfterReturning(pointcut = "controllerPointcut()", returning = "result")
public void afterControllerReturning(JoinPoint joinPoint, Object result) {
    log.info("[操作日志] 完成: {}.{} | 返回: {}",
             controller名, 方法名, 结果摘要);
}
```

**效果：** 每次用户访问任何接口，控制台自动输出：
```
[操作日志] ════════════════════════════════════════
[操作日志] 用户: admin | IP: 192.168.1.100
[操作日志] 接口: POST /api/tasks
[操作日志] 方法: TaskController.createTask
[操作日志] 时间: 2026-05-29 10:30:45
[操作日志] ════════════════════════════════════════
[操作日志] 完成: TaskController.createTask | 返回: Result(code=200, data=Task(id=1, ...))
```

**AOP 核心概念速记：**

| 术语 | 说明 | 项目中的对应 |
|------|------|-------------|
| **Aspect（切面）** | 横切关注点的模块化 | `OperationLogAspect` 类本身 |
| **JoinPoint（连接点）** | 程序执行过程中的某个点 | 所有 Controller 方法 |
| **Pointcut（切点）** | 匹配哪些 JoinPoint | `execution(* controller..*.*(..))` |
| **Advice（通知）** | 在连接点执行的动作 | `@Before`、`@AfterReturning` |
| **Weaving（织入）** | 把切面应用到目标对象的过程 | Spring 运行时通过代理织入 |

**Spring AOP 的底层实现：**

Spring AOP 是运行时代理，不是编译期织入（不像 AspectJ）。

- 如果目标类实现了接口 → 用 **JDK 动态代理**（生成一个实现了相同接口的代理对象）
- 如果目标类没实现接口 → 用 **CGLIB 代理**（生成目标类的子类）

项目中 Controller 类没有实现接口，所以用的是 **CGLIB 代理**。

## 14. MyBatis-Plus 实践

### Q：项目中有哪些 MyBatis-Plus 的高级用法？

---

#### 正式版

项目使用了 MyBatis-Plus 的四个重要特性：**LambdaQueryWrapper** 确保条件查询的类型安全（编译期检查字段名，避免字符串拼写错误）；**@TableLogic** 实现逻辑删除（查询自动拼接 `WHERE deleted=0`）；**MetaObjectHandler** 自动填充 createTime/updateTime；**分页查询配合 PageHelper**（项目中用 PageHelper 分页插件补充 MP 的分页能力）。这些都是 MP 相较于原生 MyBatis 的核心价值。

---

#### 详细版

**特性一：LambdaQueryWrapper**

```java
// 普通 QueryWrapper（字段名写字符串，容易写错）
QueryWrapper<User> wrapper = new QueryWrapper<>();
wrapper.eq("user_name", username);  // "user_name" 写错了，编译不报错，运行时才炸

// LambdaQueryWrapper（方法引用，编译期检查）
LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<>();
wrapper.eq(User::getUsername, username);  // User::getUsername，重构时 IDE 能感知
```

项目中的实际使用（`AuthService`）：
```java
long count = userMapper.selectCount(
    new LambdaQueryWrapper<User>()
        .eq(User::getUsername, registerDTO.getUsername())
);
```

**特性二：@TableLogic 逻辑删除**

```java
// User.java
@TableLogic
private Integer deleted;  // 0 = 未删，1 = 已删
```

加上这个注解后，MyBatis-Plus 会自动做三件事：
- `userMapper.deleteById(id)` → 自动转为 `UPDATE sys_user SET deleted=1 WHERE id=? AND deleted=0`
- `userMapper.selectList(wrapper)` → 自动拼接 `AND deleted=0`
- `userMapper.selectById(id)` → 自动拼接 `AND deleted=0`

不用 @TableLogic 的话，每次查询都要手动写 `WHERE deleted=0`，漏写了就可能查出已删除的数据。

**特性三：MetaObjectHandler 自动填充**

```java
@Component
public class MyMetaObjectHandler implements MetaObjectHandler {

    @Override
    public void insertFill(MetaObject metaObject) {
        // 插入时自动填充
        this.strictInsertFill(metaObject, "createTime", LocalDateTime::now, LocalDateTime.class);
        this.strictInsertFill(metaObject, "updateTime", LocalDateTime::now, LocalDateTime.class);
    }

    @Override
    public void updateFill(MetaObject metaObject) {
        // 更新时自动填充
        this.strictUpdateFill(metaObject, "updateTime", LocalDateTime::now, LocalDateTime.class);
    }
}
```

配合 Entity 中的 `@TableField(fill = FieldFill.INSERT)` 使用：
```java
@TableField(fill = FieldFill.INSERT)
private LocalDateTime createTime;

@TableField(fill = FieldFill.INSERT_UPDATE)
private LocalDateTime updateTime;
```

这样在 `userMapper.insert(user)` 时，不需要手动 `user.setCreateTime(LocalDateTime.now())`，MP 自动帮你填好。

**特性四：BaseMapper 内置方法**

项目中用到的内置方法：
```java
userMapper.selectById(id);            // 按主键查询
userMapper.selectOne(wrapper);        // 按条件查一条
userMapper.selectList(wrapper);       // 按条件查列表
userMapper.selectCount(wrapper);      // 按条件计数
userMapper.insert(user);              // 插入
userMapper.updateById(user);          // 按主键更新
userMapper.deleteById(id);            // 按主键删除（逻辑删除）
```

不需要写任何 XML，BaseMapper 提供了大部分常用方法。

## 15. 线程池配置

### Q：项目中的线程池是怎么配置的？核心参数的意义是什么？

---

#### 正式版

项目自定义了两个线程池：`taskExecutor`（核心 10，最大 20，队列 100）用于通用异步任务，`notificationExecutor`（核心 5，最大 10，队列 50）专门用于 WebSocket 推送。拒绝策略均使用 `CallerRunsPolicy`（谁提交谁执行，不丢任务），按业务类型隔离线程池避免相互影响。核心参数的工作机制是：核心线程满 → 任务进队列 → 队列满 → 创建新线程到最大数 → 超限触发拒绝策略。

---

#### 详细版

**参数详解：**

```java
@Bean("taskExecutor")
public ThreadPoolTaskExecutor taskExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(10);      // 核心线程数
    executor.setMaxPoolSize(20);       // 最大线程数
    executor.setQueueCapacity(100);    // 任务队列容量
    executor.setKeepAliveSeconds(300); // 空闲线程存活时间（5分钟）
    executor.setThreadNamePrefix("task-exec-");
    executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
    executor.initialize();
    return executor;
}
```

**任务提交后的执行流程（面试必背）：**

假设核心线程 = 10，最大 = 20，队列 = 100：

```
提交第 1 个任务 → 创建核心线程 1 去处理
提交第 2 个任务 → 创建核心线程 2 去处理
...
提交第 10 个任务 → 创建核心线程 10 去处理
提交第 11 个任务 → 核心线程都忙 → 放入队列（队列 1/100）
提交第 12-110 个任务 → 都放入队列（队列 100/100，满了）
提交第 111 个任务 → 队列也满了 → 创建非核心线程 11（当前 11/20）
提交第 112-120 个任务 → 继续创建非核心线程到 20（20/20，满了）
提交第 121 个任务 → 所有线程和队列都满了 → 触发拒绝策略
```

**拒绝策略对比：**

| 策略 | 行为 | 什么时候用 |
|------|------|-----------|
| `AbortPolicy`（默认） | 直接抛 `RejectedExecutionException` | 任务不能丢，提交方需要感知失败 |
| `CallerRunsPolicy` | 谁提交谁执行（比如 Controller 线程自己跑） | 任务不能丢，可以牺牲一点响应速度 |
| `DiscardPolicy` | 静默丢弃新任务 | 不重要的任务（如日志） |
| `DiscardOldestPolicy` | 丢弃队列中最老的任务 | 消息/通知类，新消息比旧消息重要 |

项目中用 CallerRunsPolicy 的考虑：通知可以不那么实时，但不能丢。

**为什么用两个线程池？**

如果不隔离，通知量大时会把业务线程池占满，导致创建任务、查询任务等核心业务都变慢。隔离后，各自有独立的线程资源。

**如何确定核心线程数？**

经验公式：
```
CPU 密集型（计算多）：核心 = CPU 核心数 + 1
IO 密集型（等待多）：核心 = CPU 核心数 × 2（甚至更多）
```

本项目是 Web 应用，属于 IO 密集型（大部分时间在等数据库、Redis、MQ），所以 10 个核心线程是合理的。

## 16. DTO / VO / Entity 三层分离

### Q：为什么要把请求参数、数据库实体、返回结果分成三个不同的类？

---

#### 正式版

三层分离有三个目的：**安全性**（Entity 中的 `password`、`deleted` 等字段不会意外返回给前端）、**解耦**（数据库表结构变更不影响 API 契约）、**按需传输**（DTO 可以聚合多个来源的数据，VO 可以只返回前端需要的字段）。这是分层架构中的表现层与数据层分离原则的具体实践。

---

#### 详细版

**看看如果不分离会怎样：**

```java
// 假设 User 直接被用来接收请求和返回响应
// 登录接口接收参数，用 User 接收
@PostMapping("/login")
public User login(@RequestBody User user) {
    // user.getPassword() 前端传的明文密码
}

// 但是 User 里还有 createTime、updateTime、deleted 等字段
// 前端需要传这些吗？不需要。
// 返回用户信息时把 password 也返回了，密码泄露！
```

这就是典型的安全问题+过度传输。

**项目中的三层：**

**Entity（数据层）：**
```java
// User.java — 完整映射数据库 sys_user 表
public class User {
    private Long id;
    private String username;
    private String password;    // ← 数据库存，但绝对不能返回给前端
    private String nickname;
    private String email;
    private String role;
    private Integer points;
    private Integer status;
    private Integer level;
    private String lastLoginIp;
    private LocalDateTime lastLoginTime;
    private Integer deleted;    // ← 前端不需要知道
    private LocalDateTime createTime;
    private LocalDateTime updateTime;
}
```

**DTO（请求层）：**
```java
// LoginDTO.java — 只包含登录需要的参数
public class LoginDTO {
    @NotBlank(message = "用户名不能为空")
    private String username;
    @NotBlank(message = "密码不能为空")
    private String password;
}
```

```java
// RegisterDTO.java — 注册需要的参数
public class RegisterDTO {
    @NotBlank
    private String username;
    @NotBlank
    @Size(min = 6, max = 20)
    private String password;
    @Email
    private String email;      // 注册可选填邮箱
    private String nickname;   // 注册可选填昵称
}
```

**VO（响应层）：**
```java
// LoginVO.java — 登录成功后返回给前端的数据
public class LoginVO {
    private String token;      // JWT Token
    private Long userId;
    private String username;
    private String nickname;
    private String avatar;
    private String role;
    private Integer level;
    private Integer points;
}
```

**三层之间的关系：**

```
前端请求 ──▶ DTO（接收参数 + 校验）
                    │
                    ▼
               Service（处理业务逻辑）
               Entity（操作数据库）
                    │
                    ▼
前端响应 ◀── VO（聚合返回数据）
```

Service 内部用 Entity 操作数据库，然后把数据转成 VO 返回：
```java
// AuthService 中登录成功后
return LoginVO.builder()
        .token(token)
        .role(user.getRole())
        .nickname(user.getNickname())
        .avatar(user.getAvatar())
        .userId(user.getId())
        .level(user.getLevel())
        .points(user.getPoints())
        .build();
```

## 17. 登录认证完整流程

### Q：把从用户输入密码到看到首页的整个过程串一遍

---

#### 详细版

这是面试最常见的综合题，完整流程如下：

```
浏览器                                 后端
  │                                     │
  │ 1. 输入用户名密码，点登录              │
  │ POST /api/auth/login                 │
  │ {"username":"admin","password":"123"} │
```

## 18. 项目整体架构

### Q：说说这个项目的整体架构设计

---

#### 正式版

项目采用标准的**分层架构**（表现层-业务层-数据层），后端基于 Spring Boot 3.x + JDK 17。认证授权使用 Spring Security + JWT + Redis 三件套组合，消息通知采用 RabbitMQ 解耦 + WebSocket 实时推送，数据存储用 MySQL + MyBatis-Plus，缓存和分布式锁基于 Redis + Redisson。基础设施层包含全局异常处理、参数校验、IP 限流拦截器、AOP 操作日志。项目支持角色权限控制、积分体系和仪表盘数据聚合。

#### 详细版

|------|------|------|

**分层架构图：**

```
┌─────────────────────────────────────────────────────────────┐
│  Controller 层（表现层）                                      │
│  AuthController / TaskController / DashboardController       │

│  Service 层（业务层）                                         │
│  AuthService / TaskService / DashboardService                │
│  PointsService / NotificationService / ...                   │
│  职责：业务逻辑编排、事务管理（@Transactional）                 │
├─────────────────────────────────────────────────────────────┤
│  Mapper 层（数据访问层）                                       │
│  UserMapper / TaskMapper / NotificationMapper / PointsLogMapper

└─────────────────────────────────────────────────────────────┘
```

**核心业务流（完整链路）：**

```
用户创建任务 →
  1. Controller: @Idempotent 幂等检查 → 参数校验 → 调用 Service
  2. Service: 创建任务记录到 MySQL
  3. Service: 调用 PointsService 奖励积分（@Transactional + Redisson锁）
  4. Service: 调用 RabbitMQSender 发送通知消息
  5. MQ: 消息路由到通知队列
  6. Listener: 消费消息 → 通过 WebSocket 推送给负责人
  7. 负责人不在线 → 通知存入数据库 → 下次登录拉取
```

**请求处理路径：**

```
HTTP 请求
  │
  ▼
Tomcat 接收
  │
  ▼

  │
  ▼
JwtAuthenticationFilter.doFilterInternal()  →  JWT 解析 → SecurityContext
  │
  ▼
DispatcherServlet 分发
  │
  ▼
OperationLogAspect(@Before)  →  记录操作日志
  │
  ▼
Controller 方法执行
  │
  ▼
OperationLogAspect(@AfterReturning)  →  记录返回结果
  │
  ▼
异常？→ GlobalExceptionHandler 接管 → 统一返回 Result
  │
  ▼
返回 JSON 给前端
```
