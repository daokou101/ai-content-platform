## 1. Java 基础

### 1.1 面向对象三大特性

#### 正式版

**封装**：将数据和方法绑定，对外隐藏实现细节，通过访问修饰符控制访问权限。**继承**：子类复用父类属性和方法，Java 是单继承（一个类只能有一个父类），但可实现多个接口。**多态**：同一行为在不同对象上表现不同，条件：继承/实现 + 重写 + 父类引用指向子类对象，编译看左边，运行看右边。

---

#### 详细版

**封装（Encapsulation）**
```java
public class User {
    private String password;  // 私有，外部不能直接访问

    public void setPassword(String pwd) {
        if (pwd.length() < 6) {
            throw new IllegalArgumentException("密码长度不够");
        }
        this.password = pwd;  // 通过方法控制逻辑
    }
}
```
好处：安全性（防止随意修改）、低耦合（内部变化不影响外部）、可维护性。

**继承（Inheritance）**
```java
public class Animal { public void eat() { System.out.println("吃东西"); } }
public class Dog extends Animal {
    @Override
    public void eat() { System.out.println("狗吃骨头"); }  // 重写
    public void bark() { System.out.println("汪汪"); }      // 子类特有
}
```
要点：单继承、多实现（`class Dog extends Animal implements Swimmable`）、构造方法不能继承（隐式 `super()`）。

**多态（Polymorphism）**
```java
Animal a = new Dog();
Animal b = new Cat();
a.eat();  // 狗吃骨头（实际运行 Dog 的方法）
b.eat();  // 猫吃鱼（实际运行 Cat 的方法）
```
多态的三个条件：继承（或实现接口）、方法重写（override）、父类引用指向子类对象。

### 1.2 == 和 equals 的区别

#### 正式版

`==` 比较引用类型的内存地址（基本类型比较值）。`equals` 默认也是比较地址，但 String、Integer 等类已重写为比较内容。重写 `equals` 时必须同时重写 `hashCode`——保证相等的对象有相同的哈希值，HashMap/HashSet 依赖这个约定。

---

#### 详细版

```java
String s1 = new String("abc");
String s2 = new String("abc");
System.out.println(s1 == s2);          // false，两个不同对象，地址不同
System.out.println(s1.equals(s2));     // true，String 重写了 equals，比较内容
```

**Object 的 equals 默认就是 ==：**
```java
public boolean equals(Object obj) {
    return (this == obj);
}
```

**String 的 equals 逻辑：** 先比地址（同一个对象直接 true）→ 再比类型 → 最后逐字符比较。

**为什么重写 equals 必须重写 hashCode？**

HashMap 的 put 过程：先算 hashCode 定位桶，再在桶中用 equals 找。如果两个对象 equals 相等但 hashCode 不同，会分到不同桶，HashMap 认为它们不相等——破坏约定。

```java
public class Student {
    String name;
    @Override
    public boolean equals(Object o) {
        if (o instanceof Student s) return Objects.equals(name, s.name);
        return false;
    }
    // 没有重写 hashCode！

    public static void main(String[] args) {
        Set<Student> set = new HashSet<>();
        set.add(new Student("张三"));
        set.add(new Student("张三"));
        System.out.println(set.size());  // 2（预期是 1！！！）
    }
}
```

### 1.3 String、StringBuilder、StringBuffer 的区别

#### 正式版

String 不可变（final char[]），每次修改创建新对象，线程安全（本身就是不可变对象）。StringBuilder 可变，单线程下性能最优。StringBuffer 可变，方法加了 synchronized，线程安全但略慢。单线程选 StringBuilder，多线程共享选 StringBuffer。

---

#### 详细版

**String 不可变的好处：**
- 线程安全（不需要任何同步）
- 字符串常量池可以复用（`"a"=="a"` 返回 true）
- hashCode 可以缓存（HashMap 用 String 做 key 效率高）

**String 拼接的性能问题：**
```java
String s = "a" + "b" + "c";
// 实际创建了："a"、"ab"、"abc" 三个对象（中间两个是垃圾）
```
**StringBuilder 怎么解决：**
```java
StringBuilder sb = new StringBuilder();
sb.append("a").append("b").append("c");  // 在同一个 char[] 后面追加
String result = sb.toString();
```
扩容策略：新容量 = 旧容量 × 2 + 2。

**三者选择：**

| 场景 | 推荐 |
|------|------|
| 不变的字符串 | String |
| 单线程大量拼接 | StringBuilder |
| 多线程共享变量拼接 | StringBuffer |

### 1.4 接口和抽象类的区别

#### 正式版

抽象类（`abstract class`）可有构造方法、成员变量、普通方法和抽象方法，单继承。接口（`interface`）Java 8+ 可有 default/static 方法，多实现。抽象类表达"is-a"（是什么），接口表达"ability"（能做什么）。

---

#### 详细版

| 特性 | 抽象类 | 接口 |
|------|--------|------|
| 关键字 | `abstract class` | `interface` |
| 继承/实现 | 单继承（extends） | 多实现（implements） |
| 构造方法 | 可以有 | 不能有 |
| 成员变量 | 任意类型 | 只能 `public static final` 常量 |
| 普通方法 | 可以有 | 默认/静态方法（Java 8+） |
| 设计思想 | "是什么"（is-a） | "能做什么"（ability） |

```java
// 抽象类：复用公共代码
public abstract class Animal {
    protected String name;
    public abstract void makeSound();
    public void eat() { System.out.println("吃东西"); }  // 通用行为
}
// 接口：定义能力
public interface Flyable { void fly(); }
public interface Singable { void sing(); }

public class Bird extends Animal implements Flyable, Singable {
    @Override public void makeSound() { System.out.println("啾啾"); }
    @Override public void fly() { System.out.println("飞"); }
    @Override public void sing() { System.out.println("唱歌"); }
}
```

### 1.5 重载和重写的区别

#### 正式版

重载（Overload）是同一类中方法名相同、参数列表不同，**编译时多态**。重写（Override）是子类重新定义父类方法，方法签名必须相同，返回类型可以协变（父类返回 Object 子类可返回 String），**运行时多态**。重写不能缩小访问权限（父类 protected 子类不能改成 private），不能抛出更宽泛的异常。

---

#### 详细版

```java
// 重载：同一个类，同名方法，参数不同
public class Calculator {
    public int add(int a, int b) { return a + b; }
    public int add(int a, int b, int c) { return a + b + c; }          // 参数个数不同
    public double add(double a, double b) { return a + b; }             // 参数类型不同
    // 编译时就能确定调用哪个版本
}

// 重写：子类重新实现父类方法
public class Animal {
    protected Object getInfo() throws Exception { return "Animal"; }
}
public class Dog extends Animal {
    @Override
    public String getInfo() { return "Dog"; }  
    // 返回类型：Object → String（协变，更具体）
    // 访问权限：protected → public（更宽）
    // 异常：throws Exception → 无（更窄）
}
```

| 特性 | 重载（Overload） | 重写（Override） |
|------|-----------------|-----------------|
| 发生位置 | 同一个类 | 子类和父类 |
| 方法名 | 相同 | 相同 |
| 参数列表 | 必须不同 | 必须相同 |
| 返回类型 | 可不同 | 必须相同（或协变） |
| 访问权限 | 无限制 | 不能缩小 |
| 异常 | 无限制 | 不能抛出更宽的异常 |
| 多态类型 | 编译时多态 | 运行时多态 |

### 1.6 Java 异常体系

#### 正式版

Java 异常分为 **运行时异常（RuntimeException）** 和 **受检异常（Checked Exception）**。运行时异常包括 NullPointerException、ArrayIndexOutOfBoundsException、ClassCastException 等，编译器不强制处理。受检异常包括 IOException、SQLException 等，必须 try-catch 或 throws 声明。Error 表示严重系统问题（如 OutOfMemoryError、StackOverflowError），不建议捕获。

---

#### 详细版

**异常体系结构：**
```
Throwable
├── Error（系统级错误，不可恢复）
│   ├── OutOfMemoryError（OOM）
│   ├── StackOverflowError（栈溢出）
│   └── NoClassDefFoundError
│
└── Exception（程序级异常）
    ├── RuntimeException（运行时异常，不强制处理）
    │   ├── NullPointerException（NPE，最常见）
    │   ├── ArrayIndexOutOfBoundsException（数组越界）
    │   ├── ClassCastException（类型转换异常）
    │   ├── IllegalArgumentException（非法参数）
    │   └── ArithmeticException（算术异常，如除0）
    │
    └── 受检异常（必须处理）
        ├── IOException（IO 异常）
        ├── SQLException（数据库异常）
        ├── ClassNotFoundException（类未找到）
        └── FileNotFoundException（文件未找到）
```

**受检异常的处理：**
```java
// 方式一：try-catch
try {
    FileInputStream fis = new FileInputStream("test.txt");
} catch (FileNotFoundException e) {
    e.printStackTrace();
}

// 方式二：方法声明 throws
public void readFile() throws FileNotFoundException {
    FileInputStream fis = new FileInputStream("test.txt");
}
```

**运行时异常不需要处理：**
```java
public void setName(String name) {
    if (name == null) {
        throw new IllegalArgumentException("name 不能为 null");  // 运行时异常
    }
}
// 调用方可以不处理，让程序自然终止或由全局异常处理器统一处理
```

**最佳实践：**
- 项目中用 `GlobalExceptionHandler` 统一拦截异常，不散落在每个 Controller 中
- 自定义业务异常 `BusinessException` 继承 RuntimeException，避免每个方法都 throws
- finally 块中不要 return 或抛异常（会覆盖 try 块的返回值）

### 1.7 泛型（类型擦除）

#### 正式版

Java 泛型是**编译期**的机制，底层通过**类型擦除**实现。编译时泛型类型被擦除为上限类型（如 `List<T>` 擦除为 `List<Object>`，`List<? extends Number>` 擦除为 `List<Number>`），字节码中没有泛型信息。类型擦除导致：不能 `new T()`、不能 `new T[]`、不能 `instanceof` 泛型类型、静态字段不共享泛型类型信息。

---

#### 详细版

**泛型的作用：**
```java
// 没有泛型：需要强制类型转换，不安全
List list = new ArrayList();
list.add("hello");
String s = (String) list.get(0);  // 需要强转

// 有泛型：编译期类型检查，不用强转
List<String> list = new ArrayList<>();
list.add("hello");
String s = list.get(0);  // 不需要强转
```

**类型擦除的过程：**
```java
// 源码中
public class Box<T> {
    private T value;
    public T get() { return value; }
    public void set(T value) { this.value = value; }
}

// 编译后（类型擦除后，相当于）
public class Box {
    private Object value;  // T → Object（因为 T 没有指定上限）
    public Object get() { return value; }
    public void set(Object value) { this.value = value; }
}

// 如果指定了上限
public class Box<T extends Number> {
    private T value;
    // 编译后：T → Number（不是 Object）
    private Number value;
}
```

**类型擦除导致的问题：**
```java
// 1. 不能 new T()
public class Box<T> {
    private T value;
    public void create() {
        value = new T();  // 编译错误！不知道 T 是哪个类
    }
}

// 2. 不能 new T[]
public Box<T>[] createArray() {
    return new Box<T>[10];  // 编译错误！泛型数组
}

// 3. 不能 instanceof
if (value instanceof T) { }  // 编译错误！T 已被擦除

// 4. 不能重载（因为擦除后方法签名相同）
public void process(List<String> list) { }
public void process(List<Integer> list) { }  // 编译错误！都是 List
```

**通配符：**
```java
// ? extends T：生产者，只能读不能写
List<? extends Number> list = new ArrayList<Integer>();
Number n = list.get(0);  // OK，读到的是 Number
list.add(1);              // 编译错误！不知道具体类型

// ? super T：消费者，只能写不能读
List<? super Integer> list = new ArrayList<Number>();
list.add(1);              // OK，可以写 Integer
Integer n = list.get(0);  // 编译错误！读出来的是 Object

// ?：无限定通配符
List<?> list = new ArrayList<String>();
list.add("hello");        // 编译错误！不能写（null 除外）
Object o = list.get(0);   // 只能读到 Object
```

**PECS 原则（Producer Extends, Consumer Super）：**
- 要从集合中读取数据（生产者）：用 `? extends`
- 要向集合中写入数据（消费者）：用 `? super`
- 读写都有：不用通配符

### 1.8 反射机制

#### 正式版

反射是在运行时获取类的信息并操作类（构造方法、字段、方法）的机制。核心 API：`Class.forName()`、`getDeclaredFields()`、`invoke()`、`setAccessible()`。Java 反射可以绕过 private 访问限制。Spring 框架大量使用反射（IoC 创建 Bean、DI 注入字段、AOP 生成代理）。反射性能比直接调用慢，内部调用有缓存优化。

---

#### 详细版

**获取 Class 对象的三种方式：**
```java
// 方式一：Class.forName（最常用，用在框架中）
Class<?> clazz = Class.forName("com.example.User");

// 方式二：类名.class
Class<User> clazz = User.class;

// 方式三：对象.getClass()
User user = new User();
Class<?> clazz = user.getClass();
```

**反射操作：**
```java
public class User {
    private String name;
    public User() {}
    public User(String name) { this.name = name; }
    private String getName() { return name; }
}

// 反射创建对象
Class<?> clazz = Class.forName("com.example.User");
User user = (User) clazz.getDeclaredConstructor().newInstance();

// 反射调用私有方法
Method method = clazz.getDeclaredMethod("getName");
method.setAccessible(true);       // 绕过 private 检查
String name = (String) method.invoke(user);

// 反射修改私有字段
Field field = clazz.getDeclaredField("name");
field.setAccessible(true);
field.set(user, "张三");

// 反射调用带参构造
Constructor<?> constructor = clazz.getDeclaredConstructor(String.class);
User user2 = (User) constructor.newInstance("李四");
```

**Spring 中的反射：**
- `@ComponentScan`扫描包 → 用 `Class.forName` 加载类
- `@Autowired` → 找到字段类型，`field.set(bean, dependencyBean)`
- AOP → 运行时用反射读取注解信息，生成代理对象

### 1.9 Java 8 新特性

#### 正式版

核心 5 大特性：**Lambda 表达式**（简化匿名内部类）、**Stream API**（函数式操作集合）、**Optional**（优雅处理 null）、**函数式接口**（@FunctionalInterface）、**新的日期时间 API**（LocalDate、LocalDateTime，线程安全）。还有接口的 default/static 方法和方法引用（::）。

```java
// Lambda 写法
new Thread(() -> System.out.println("hello")).start();
```

```java
    .average()
    .orElse(0);
```

**Optional：**
```java

public String getCity(User user) {
```

### 1.10 BIO / NIO / AIO 的区别

#### 正式版

**BIO**（同步阻塞）：一个连接一个线程，连接建立后线程一直阻塞等待数据。**NIO**（同步非阻塞）：基于 Channel + Buffer + Selector，一个线程管理多个连接，Selector 轮询哪些 Channel 有数据可读/写。**AIO**（异步非阻塞）：操作系统完成后通知应用，应用不需要轮询。Java 中 BIO 对应 `Socket/ServerSocket`，NIO 对应 `SocketChannel/Selector`，AIO 对应 `AsynchronousSocketChannel`。Tomcat 传统使用 BIO，NIO 模式下支持更高并发。

## 2. 集合框架

### 2.1 ArrayList 和 LinkedList

#### 正式版

ArrayList 基于动态数组，初始容量 10，扩容 1.5 倍。随机访问 O(1)，尾部插入 O(1) 均摊，中间插入/删除 O(n)。LinkedList 基于双向链表，随机访问 O(n)，头尾操作 O(1)，中间插入/删除 O(n)（需要先遍历找到位置）。实际开发中大部分场景用 ArrayList，除非在头部频繁增删且数据量大。

---

#### 详细版

**ArrayList 扩容过程：**
```java
// 1. new ArrayList() → 空 Object[]
// 2. 第一次 add → 扩容到 10
// 3. 第 11 次 add → 扩容到 15（1.5 倍）
// 4. 扩容 = 创建新数组 + 全部复制过去（System.arraycopy）
```

**LinkedList 节点结构：**
```java
private static class Node<E> {
    E item;          // 数据
    Node<E> next;    // 后驱指针
    Node<E> prev;    // 前驱指针
}
// 头尾操作只需改指针，中间操作需要遍历
```

**性能对比真实场景：**
```java
// ArrayList 适合：随机读取多
for (int i = 0; i < list.size(); i++) { list.get(i); }  // O(1)

// LinkedList 适合：双端队列场景
list.addFirst(x);   // O(1)
list.removeLast();  // O(1)

// 注意：中间插入 LinkedList 不一定比 ArrayList 快
list.add(1000, x);  // ArrayList: 移动 1000 之后的元素 O(n)
                    // LinkedList: 遍历到 1000 的位置 O(n) + 改指针 O(1)
```

### 2.2 HashMap 底层实现（完整版，高频）

#### 正式版

JDK 1.8 的 HashMap 结构为**数组 + 链表 + 红黑树**。put 过程：计算 `hash(key)` → `(n-1) & hash` 定位桶 → 无冲突直接插 → 有冲突用链地址法 → 链表长 ≥8 且数组长 ≥64 转红黑树（O(n)→O(log n)）。默认加载因子 0.75，扩容为 2 倍并 rehash。**线程不安全**，多线程用 ConcurrentHashMap。

---

#### 详细版

**数据结构图：**
```
table (数组)
┌────┐
│ 0  │ → null
├────┤
│ 1  │ → Node("A") → Node("B") → Node("C")  [链表，哈希冲突]
├────┤
│ 2  │ → null
├────┤
│ 3  │ → TreeNode("X") → TreeNode("Y") [红黑树，链长 ≥8 且数长 ≥64]
├────┤
│... │
└────┘
```

**put 方法的完整过程（面试必背）：**
```java
map.put("name", "张三");

// 第一步：计算哈希值
int hash = ("name").hashCode() ^ (hash >>> 16);  // 扰动函数，高位参与运算

// 第二步：计算桶的位置（等价于 hash % n，但位运算更快）
int index = (table.length - 1) & hash;  // table.length 必须是 2 的幂

// 第三步：确定桶位置后
// 情况 1：桶为 null → 直接 new Node 放入
// 情况 2：桶不为 null → 遍历链表/红黑树
//   情况 2.1：找到 key 相同 → 覆盖旧 value
//   情况 2.2：没找到 → 追加到链表末尾（尾插法，JDK 1.8 改的）
//      → 检查链表长度，≥8 且数组长度 ≥64 则转红黑树

// 第四步：检查是否需要扩容
// if (size > threshold) → resize() （threshold = capacity × loadFactor）
```

**为什么初始容量 16、加载因子 0.75？**

16 是 2 的 4 次方（为了 `(n-1) & hash` 生效），0.75 是时间和空间的平衡点。太小（0.5）浪费空间，太大（0.9）冲突多查询慢。扩容阈值 = 16 × 0.75 = 12，第 13 个元素插入时扩容。

**扩容（resize）过程：**
```java
// 旧容量 16 → 新容量 32
// 创建新数组 → 遍历旧数组每个链表/树 → 重新计算每个元素的位置
// 为什么要重新计算？因为 n 变了，(n-1) & hash 的结果可能不同
// JDK 1.8 优化：元素在新的位置要么不变，要么在"原位置 + 旧容量"处
// 不需要重新计算 hash，只需要看 hash 的新增 bit 是 0 还是 1
```

**JDK 1.7 vs 1.8 的 HashMap：**

| 特性 | JDK 1.7 | JDK 1.8 |
|------|---------|---------|
| 数据结构 | 数组 + 链表 | 数组 + 链表 + 红黑树 |
| 插入方式 | 头插法（并发可能死循环） | 尾插法 |
| 哈希计算 | 4 次位运算 + 5 次异或 | 1 次高位异或 |
| 扩容后位置 | 全部重新计算 | 原位置或原位置+旧容量 |

### 2.3 ConcurrentHashMap

#### 正式版

JDK 1.8 的 ConcurrentHashMap 采用 **CAS + synchronized** 实现线程安全。put 时桶为 null 用 CAS 无锁写入，桶不为 null 用 synchronized 锁桶（锁链表头/树根节点），锁粒度是单个桶，并发度远高于 Hashtable（锁整张表）。get 完全无锁（Node 的 value 和 next 用 volatile 保证可见性）。迭代器是弱一致性的，不抛 ConcurrentModificationException。

---

#### 详细版

**put 过程（JDK 1.8）：**
```java
public V put(K key, V value) {
    // 计算 hash
    // 死循环 + CAS（自旋）
    for (;;) {
        // 情况 1：table 还没初始化 → 初始化（CAS 保证只有一个线程初始化）
        // 情况 2：桶为空 → CAS 无锁插入（期望值 null，新值 Node）
        //   成功 → 结束
        //   失败 → 重试（其他线程先插入了）
        // 情况 3：桶不为空 → synchronized 锁桶
        //   锁住链表头节点或红黑树根节点
        //   遍历插入或覆盖
        // 情况 4：正在扩容 → 帮忙扩容
    }
}
```

**get 为什么不用锁：**
```java
public V get(Object key) {
    // 直接计算 hash → 找到桶 → 遍历
    // Node 的 value 和 next 都声明为 volatile
    // volatile 保证一个线程修改了 value，其他线程立即看到
}

transient volatile Node<K,V>[] table;  // 数组 volatile
static class Node<K,V> {
    volatile V val;      // value 可见性
    volatile Node<K,V> next;  // next 指针可见性
}
```

**并发度对比：**

| 集合 | 锁粒度 | 并发读 | 并发写 |
|------|--------|--------|--------|
| Hashtable | 整张表 | 互斥 | 互斥 |
| ConcurrentHashMap JDK 1.7 | Segment（默认 16 个） | 不互斥 | 不同 Segment 互不干扰 |
| ConcurrentHashMap JDK 1.8 | 单个桶（可到几百个） | 完全不需锁 | 不同桶互不干扰 |

### 2.4 HashSet 去重原理

#### 正式版

HashSet 底层是 HashMap，`add(e)` 实际上是 `map.put(e, PRESENT)`，PRESENT 是一个固定的 Object 占位对象。HashMap put 的返回值是旧 value（第一次为 null），HashSet 根据返回值是否为 null 判断是否重复。所以 HashSet 判断重复的依据是元素的 `hashCode()` 和 `equals()` 方法。

---

### 2.5 TreeMap / TreeSet 底层（红黑树）

#### 正式版

TreeMap 底层是**红黑树**（自平衡二叉搜索树），插入/删除/查找均为 O(log n)。元素按 key 的自然顺序或 Comparator 排序。TreeSet 底层是 TreeMap，`add(e)` 相当于 `map.put(e, PRESENT)`。红黑树的约束：节点非红即黑、根是黑、叶是黑、红节点的子节点为黑、从根到叶的每条路径黑色节点数相同。

---

#### 详细版

**使用场景：**
```java
// 需要有序的场景
TreeMap<String, Integer> map = new TreeMap<>();
map.put("b", 2);
map.put("a", 1);
map.put("c", 3);
// 遍历时按字母顺序：a→b→c
for (Map.Entry<String, Integer> entry : map.entrySet()) {
    System.out.println(entry.getKey());  // a, b, c
}

// 自定义排序
TreeMap<Integer, String> descMap = new TreeMap<>((a, b) -> b - a);

// 常用方法
map.firstKey();     // 最小 key
map.lastKey();      // 最大 key
map.lowerKey(k);    // 小于 k 的最大 key
map.higherKey(k);   // 大于 k 的最小 key
map.subMap(from, to);  // 范围查询
```

**HashMap vs TreeMap：**

| | HashMap | TreeMap |
|--|---------|---------|
| 底层结构 | 数组+链表+红黑树 | 红黑树 |
| 时间复杂度 | O(1)（最好）~ O(log n)（红黑树部分） | O(log n)（所有操作） |
| 顺序 | 无序 | 有序（自然顺序或 Comparator） |
| null | key/value 都可以为 null | key 不能为 null（需要比较） |
| 适用场景 | 大部分场景 | 需要排序或范围查询 |

### 2.6 LinkedHashMap 和 LRU 缓存

#### 正式版

LinkedHashMap 继承 HashMap，内部额外维护一个双向链表记录元素顺序（默认插入顺序或访问顺序）。`accessOrder=true` 时开启访问顺序（每次 get/put 将元素移到链表末尾），基于此可方便实现 **LRU 缓存**：重写 `removeEldestEntry()` 方法，在插入新元素时检查是否超过容量，超过则移除链表头部（最久未访问的）。

---

#### 详细版

**实现 LRU 缓存：**
```java
public class LRUCache<K, V> extends LinkedHashMap<K, V> {
    private final int maxCapacity;

    public LRUCache(int maxCapacity) {
        super(maxCapacity, 0.75f, true);  // accessOrder = true
        this.maxCapacity = maxCapacity;
    }

    // 每次 put 后自动调用，返回 true 就移除最老的条目
    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > maxCapacity;
    }

    public static void main(String[] args) {
        LRUCache<Integer, String> cache = new LRUCache<>(3);
        cache.put(1, "a");  // 队列: 1
        cache.put(2, "b");  // 队列: 1, 2
        cache.put(3, "c");  // 队列: 1, 2, 3
        cache.get(1);       // 访问 1 → 1 移到末尾: 2, 3, 1
        cache.put(4, "d");  // 超过容量，移除头部 2: 3, 1, 4
        System.out.println(cache.keySet());  // [3, 1, 4]（2 被移除了）
    }
}
```

### 2.7 集合的 fail-fast 机制

#### 正式版

fail-fast（快速失败）是 Java 集合的一种错误检测机制。当迭代器遍历集合时，如果集合结构被修改（增删元素），迭代器会立刻抛出 `ConcurrentModificationException`，而不是在后续某个不确定的时间出问题。原理：迭代器内部维护一个 `modCount` 计数器，每次修改集合时 modCount++，迭代时检查 modCount 是否被修改。注意：`ConcurrentHashMap` 的迭代器是**弱一致性**的，不抛此异常。

---

#### 详细版

```java
List<String> list = new ArrayList<>(Arrays.asList("a", "b", "c", "d"));
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    String s = it.next();
    if (s.equals("b")) {
        list.remove(s);  // 集合被修改了，modCount++
        // 下次调用 it.next() 时，检查 modCount 不一致 → 抛异常！
    }
}
// 抛出 ConcurrentModificationException
```

**正确的删除方式：**
```java
// 方式一：用迭代器的 remove
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    String s = it.next();
    if (s.equals("b")) {
        it.remove();  // OK，迭代器的 remove 会同步更新 modCount
    }
}

// 方式二：用 Collection.removeIf（Java 8+）
list.removeIf(s -> s.equals("b"));

// 方式三：用普通的 for 循环倒着删
for (int i = list.size() - 1; i >= 0; i--) {
    if (list.get(i).equals("b")) list.remove(i);
}
```

## 3. 并发编程

### 3.1 synchronized 原理（锁升级）

#### 正式版

synchronized 基于 Monitor 机制，JDK 1.6 后引入锁升级：**无锁 → 偏向锁（一个线程反复获取，记录线程 ID）→ 轻量级锁（少量竞争，CAS 自旋）→ 重量级锁（大量竞争，线程挂起）**。锁信息存在 Java 对象头的 Mark Word 中。偏向锁和轻量级锁是用户态操作，重量级锁涉及用户态到内核态的切换。

---

#### 详细版

**三种用法：**
```java
// 1. 修饰实例方法 → 锁 this
public synchronized void method() { }
// 2. 修饰静态方法 → 锁 Class 对象
public static synchronized void method() { }
// 3. 修饰代码块 → 锁指定对象
public void method() {
    synchronized (this) { }
}
```

**对象头 Mark Word（32 位 JVM）：**
```
无锁：    | unused:25 | hash:4 | age:1 | 0 1 |
偏向锁：  | thread:23 | epoch | age | 0 1 |
轻量级锁：| ptr_to_lock_record | 0 0 |
重量级锁：| ptr_to_heavy_monitor | 1 0 |
```

**锁升级过程（面试必背）：**

```
新创建的对象 → 可偏向状态
    │
    ├── 线程 A 获取锁
    │   → Mark Word 记录线程 A 的 ID（偏向锁）
    │   → 线程 A 下次再来：直接比对 ID，一样就执行，不用 CAS
    │
    ├── 线程 B 尝试获取同一把锁（竞争发生）
    │   → 偏向锁撤销
    │   → 升级为轻量级锁
    │   → 线程 A 和 B 通过 CAS 自旋竞争锁
    │   → 自旋：不挂起线程，空转 CPU 等待
    │
    └── 自旋超过一定次数（或自旋线程太多）
        → 升级为重量级锁
        → 没拿到锁的线程挂起（阻塞，进入等待队列）
        → 涉及用户态 → 内核态切换（开销最大）
```

**为什么 JDK 1.6 做锁优化？**

早期 synchronized 直接就是重量级锁，线程挂起/恢复需要操作系统帮忙，用户态切换到内核态，开销很大。现实中大部分代码的锁竞争并不激烈，甚至根本没有竞争。所以 JDK 1.6 引入偏向锁和轻量级锁，让"没有竞争的锁"和"低竞争的锁"开销降到最低。

### 3.2 volatile 关键字

#### 正式版

volatile 两个作用：**保证可见性**（一个线程修改 volatile 变量，其他线程立即可见，底层通过缓存一致性协议 MESI 实现）和**禁止指令重排序**（读写前后插入内存屏障）。volatile **不保证原子性**，`count++` 这种复合操作仍需加锁。经典应用：双重检查锁中的状态标志、单例模式的 `instance` 字段。

---

### 3.3 线程池核心参数

#### 正式版

ThreadPoolExecutor 7 大参数：**corePoolSize**、**maximumPoolSize**、**keepAliveTime**、**unit**、**workQueue**、**threadFactory**、**handler**。执行流程：核心线程 → 任务队列 → 非核心线程 → 拒绝策略。拒绝策略有 AbortPolicy（抛异常）、CallerRunsPolicy（提交方自己跑）、DiscardPolicy（丢弃）、DiscardOldestPolicy（丢弃最老）。阿里规范要求手动用 ThreadPoolExecutor 创建，不用 Executors（可能 OOM）。

---

#### 详细版

**执行流程（面试必背）：**
```java
// 配置：core=2, max=5, queue=10

// 任务 1 → 创建核心线程 1
// 任务 2 → 创建核心线程 2
// 任务 3 → 核心线程都忙 → 进队列（队列 1/10）
// 任务 4-12 → 继续进队列（队列 10/10，满了）
// 任务 13 → 队列满了 → 创建非核心线程 3（当前 3/5）
// 任务 14-15 → 创建非核心线程 4、5（当前 5/5 最大值）
// 任务 16 → 所有都满了 → 拒绝策略
```

**为什么不推荐 Executors：**
```java
// newFixedThreadPool → 无界队列 LinkedBlockingQueue（最大 Integer.MAX_VALUE）
// → 任务堆积可能导致 OOM
Executors.newFixedThreadPool(10);  // 不推荐！

// newCachedThreadPool → 最大线程数 Integer.MAX_VALUE
// → 线程创建太多导致 OOM
Executors.newCachedThreadPool();  // 不推荐！

// 正确的做法：
new ThreadPoolExecutor(10, 20, 60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(100),  // 有界队列！限制了堆积量
    new ThreadPoolExecutor.CallerRunsPolicy());
```

### 3.4 ThreadLocal

#### 正式版

ThreadLocal 为每个线程提供独立的变量副本。底层：每个 Thread 维护一个 ThreadLocalMap，key 是 ThreadLocal 对象（**弱引用**），value 是变量值。**必须用完后调用 remove()**，否则线程池场景下线程复用可能导致内存泄漏（key 被 GC 回收了，value 还强引用着）。

---

#### 详细版

**场景：SimpleDateFormat 线程不安全**
```java
// 问题：多个线程共用同一个 SimpleDateFormat
private static final ThreadLocal<SimpleDateFormat> LOCAL =
    ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));

public static String format(Date date) {
    return LOCAL.get().format(date);  // 每个线程有自己的实例
}
```

**内存泄漏分析（面试高频）：**
```
Thread 存活（线程池的线程长期存活）
  │
  └── ThreadLocalMap
      │
      ├── Entry 1: key=弱引用(ThreadLocal@123), value="hello"
      │             如果 ThreadLocal 外部强引用断了（tl = null）
      │             GC 时 key 被回收 → key=null
      │             value 还是强引用 → 永远无法回收 → 内存泄漏！
      │
      └── Entry 2: ...
```

**所以：`tl.remove()` 一定要在 finally 中调用！**
```java
ThreadLocal<String> tl = new ThreadLocal<>();
try {
    tl.set("hello");
    // ... 使用
} finally {
    tl.remove();  // 不删就泄漏！
}
```

**项目中的应用：**
- `SecurityContextHolder.getContext().getAuthentication()` → 底层用 ThreadLocal 存当前请求的用户信息
- `RequestContextHolder.getRequestAttributes()` → 底层用 ThreadLocal 存当前请求的 Request 对象

### 3.5 CAS 和 ABA 问题

#### 正式版

CAS（Compare And Swap）是一种无锁原子操作：比较内存值 V 和预期值 A，相等则更新为新值 B，不相等则重试（自旋）。Java 中 `AtomicInteger`、`AtomicStampedReference` 等底层用 `Unsafe.compareAndSwapInt` 实现。CAS 有三个问题：**ABA 问题**（值从 A 变成 B 又变回 A，CAS 检测不到变化）、**循环开销大**（长时间自旋浪费 CPU）、**只能保证一个变量的原子性**。

---

#### 详细版

**ABA 问题详解：**
```java
// 场景：栈操作
// 初始：栈顶 A → B → C
// 线程 1：想 pop，CAS 比较栈顶 = A？
// 线程 2：pop A → pop B → push A（栈变回 A → C）
// 线程 1：CAS 发现栈顶还是 A → 成功！但 A 下面的已经不是 B 了！
// 数据错误！

// 解决：AtomicStampedReference（带版本号的 CAS）
AtomicStampedReference<String> ref = new AtomicStampedReference<>("A", 0);
int[] stampHolder = new int[1];
String value = ref.get(stampHolder);  // value="A", stamp=0
// CAS 时不仅要比较值，还要比较版本号
ref.compareAndSet("A", "B", stampHolder[0], stampHolder[0] + 1);
```

**自旋开销问题：**
```java
// CAS 是"自旋"的——失败了就在循环里重试
public final int getAndAddInt(Object o, long offset, int delta) {
    int v;
    do {
        v = getIntVolatile(o, offset);  // 读取当前值
    } while (!compareAndSwapInt(o, offset, v, v + delta));
    // 如果 CAS 失败（其他线程修改了），重新读再试
    // 如果竞争激烈，可能自旋很多次 → 浪费 CPU
}
```

### 3.6 AQS（AbstractQueuedSynchronizer）原理

#### 正式版

AQS 是 JUC 锁和同步器的基类（ReentrantLock、CountDownLatch、Semaphore 等底层都基于 AQS）。核心是 **volatile int state**（同步状态）和 **CLH 双向队列**（等待线程队列）。独占模式（如 ReentrantLock）：`tryAcquire()` 尝试获取 state，失败则入队等待。共享模式（如 Semaphore）：`tryAcquireShared()` 返回剩余许可数。AQS 使用模板方法模式，子类实现 `tryAcquire`/`tryRelease` 等抽象方法。

---

#### 详细版

**AQS 的核心结构：**
```java
public abstract class AbstractQueuedSynchronizer {
    // 同步状态（volatile 保证可见性）
    private volatile int state;

    // CLH 双向队列（等待获取锁的线程队列）
    private transient volatile Node head;
    private transient volatile Node tail;

    // 队列节点（每个节点包装一个等待的线程）
    static final class Node {
        volatile Thread thread;       // 等待的线程
        volatile Node prev;           // 前驱节点
        volatile Node next;           // 后继节点
        volatile int waitStatus;      // 等待状态
    }
}
```

**独占模式（ReentrantLock）的工作原理：**
```java
// lock.lock() 的调用过程
// 1. tryAcquire(1)：尝试获取锁
//     - state == 0 → CAS(state, 0, 1) 获取成功
//     - 当前线程是锁的持有者 → state + 1（可重入）
// 2. 获取失败 → addWaiter() 将当前线程包装为 Node 入队
// 3. acquireQueued()：自旋 + 阻塞
//     - 检查前驱是不是 head → 是就再 tryAcquire
//     - 不是或被唤醒 → 检查前驱的 waitStatus，挂起线程

// unlock() 的调用过程
// 1. tryRelease(1)：释放锁
//     - state - 1
//     - state == 0 → 锁完全释放，owner = null
// 2. unparkSuccessor(head)：唤醒 head 的下一个节点线程
```

---

### 3.7 ReentrantLock 和 synchronized 对比

#### 正式版

共同点：都是可重入锁（同一个线程可以多次获取同一把锁）。区别：synchronized 是关键字，自动加锁解锁；ReentrantLock 是 API，需手动 lock/unlock（推荐在 finally 中 unlock）。ReentrantLock 支持**尝试获取锁**（tryLock）、**可中断**（lockInterruptibly）、**公平锁**（fair=true）、**绑定多个 Condition**（精细控制线程唤醒）。

---

#### 详细版

```java
// synchronized（自动释放，简单）
synchronized (lock) {
    // 业务逻辑
}  // 出作用域自动解锁

// ReentrantLock（灵活但必须手动解锁）
ReentrantLock lock = new ReentrantLock(true);  // true = 公平锁
lock.lock();
try {
    // 业务逻辑
} finally {
    lock.unlock();  // 必须自己释放！
}
```

**ReentrantLock 特有的功能：**
```java
// 1. tryLock：尝试获取锁，拿不到不阻塞
if (lock.tryLock(3, TimeUnit.SECONDS)) {
    try { /* ... */ } finally { lock.unlock(); }
} else {
    System.out.println("获取锁失败，做其他处理");
}

// 2. lockInterruptibly：可中断
lock.lockInterruptibly();  // 其他线程中断当前线程时抛出 InterruptedException

// 3. 公平锁：按请求顺序获取锁（synchronized 是非公平的）
new ReentrantLock(true);  // 公平

// 4. Condition：精细控制
Condition notFull = lock.newCondition();
Condition notEmpty = lock.newCondition();
// notEmpty.await()   → 队列为空时等待
// notEmpty.signal()  → 有数据时唤醒等待的消费者
```

### 3.8 CountDownLatch / CyclicBarrier / Semaphore

#### 正式版

**CountDownLatch**（倒计时器）：一个线程等待多个线程执行完毕。`countDown()` 减计数，`await()` 阻塞直到计数为 0。一次性，计数归零后不可重置。**CyclicBarrier**（循环屏障）：多个线程互相等待，都到达屏障后同时执行。可重置（循环使用）。支持 barrierAction（到达屏障后执行的线程）。**Semaphore**（信号量）：控制同时访问资源的线程数，`acquire()` 获取许可，`release()` 释放许可，可做限流。

---

#### 详细版

**CountDownLatch 示例：**
```java
// 主线程等待所有子任务完成
CountDownLatch latch = new CountDownLatch(5);

for (int i = 0; i < 5; i++) {
    new Thread(() -> {
        System.out.println(Thread.currentThread().getName() + " 执行完毕");
        latch.countDown();  // 计数 -1
    }).start();
}

latch.await();  // 主线程阻塞，直到计数为 0
System.out.println("所有子任务完成，主线程继续");
```

**CyclicBarrier 示例：**
```java
// 5 个玩家都加载完毕才开赛
CyclicBarrier barrier = new CyclicBarrier(5, () -> {
    System.out.println("所有人都准备好了，开始！");
});

for (int i = 0; i < 5; i++) {
    new Thread(() -> {
        System.out.println(Thread.currentThread().getName() + " 加载完成");
        barrier.await();  // 等待其他玩家
        System.out.println(Thread.currentThread().getName() + " 进入游戏");
    }).start();
}
// 可以重复使用（barrier.reset()）
```

**Semaphore 示例：**
```java
// 最多 3 个线程同时访问
Semaphore semaphore = new Semaphore(3);

for (int i = 0; i < 10; i++) {
    new Thread(() -> {
        semaphore.acquire();  // 获取许可（没拿到就阻塞）
        System.out.println(Thread.currentThread().getName() + " 开始执行");
        Thread.sleep(1000);
        semaphore.release();  // 释放许可
    }).start();
}
```

| 同步器 | 核心用途 | 可重置 |
|--------|---------|--------|
| CountDownLatch | 一个线程等 N 个线程完成 | ❌ |
| CyclicBarrier | N 个线程互相等待到齐 | ✅ |
| Semaphore | 控制并发访问的线程数 | ✅ |

### 3.9 并发容器

#### 正式版

并发场景下的容器选择：**CopyOnWriteArrayList** 替代 ArrayList（读多写少场景，写时复制原数组，读不加锁）。**ConcurrentHashMap** 替代 HashMap。**BlockingQueue**（阻塞队列，含 ArrayBlockingQueue/LinkedBlockingQueue 等）用于生产者-消费者模式。**ConcurrentLinkedQueue** 高性能无锁队列（CAS 实现）。

---

#### 详细版

**CopyOnWriteArrayList：**
```java
// 读操作完全不加锁（读的是快照）
public E get(int index) { return elementData[index]; }

// 写操作：加锁 + 复制新数组 + 替换引用
public boolean add(E e) {
    synchronized (lock) {
        Object[] old = getArray();
        Object[] newArr = Arrays.copyOf(old, old.length + 1);
        newArr[old.length] = e;
        setArray(newArr);  // 替换引用（volatile 保证可见性）
        return true;
    }
}
// 缺点：每次写都复制整个数组，内存开销大。适合读多写少（如黑名单、配置）
```

**BlockingQueue（阻塞队列）：**
```java
// 生产者-消费者模式的核心
BlockingQueue<Task> queue = new ArrayBlockingQueue<>(100);

// 生产者
queue.put(task);  // 队列满时阻塞等待

// 消费者
Task task = queue.take();  // 队列空时阻塞等待

// 常用实现：
// ArrayBlockingQueue       → 有界，数组，公平性可配
// LinkedBlockingQueue      → 可选有界，链表
// PriorityBlockingQueue    → 优先级排序
// SynchronousQueue         → 容量 0，直接传递（线程池 newCachedThreadPool 用这个）
// DelayQueue               → 延迟执行
```

### 3.10 CompletableFuture

#### 正式版

CompletableFuture 是 Java 8 引入的异步编程工具，实现了 CompletionStage 接口，支持声明式编排异步任务。核心 API：**supplyAsync**（异步执行有返回值）、**thenApply**（串行转换）、**thenCompose**（串联两个异步任务）、**thenCombine**（合并两个异步结果）、**allOf**（等待所有完成）、**anyOf**（任意一个完成即返回）。默认使用 ForkJoinPool.commonPool()，可自定义线程池。

---

#### 详细版

```java
// 自定义线程池
ExecutorService executor = Executors.newFixedThreadPool(10);

// 异步执行任务
CompletableFuture<User> future = CompletableFuture.supplyAsync(() -> {
    return userService.findById(1L);
}, executor);

// 串行：查用户 → 查用户的订单
CompletableFuture<List<Order>> future = CompletableFuture
    .supplyAsync(() -> userService.findById(1L))
    .thenCompose(user -> orderService.findByUserId(user.getId()));

// 并行：同时查用户信息和商品信息，合并结果
CompletableFuture<User> userFuture = CompletableFuture.supplyAsync(() -> userService.findById(1L));
CompletableFuture<Product> productFuture = CompletableFuture.supplyAsync(() -> productService.findById(1L));

CompletableFuture<String> combined = userFuture.thenCombine(productFuture, (user, product) -> {
    return user.getName() + " 购买了 " + product.getName();
});

// 等待所有完成
CompletableFuture.allOf(userFuture, productFuture).join();

// 异常处理
CompletableFuture.supplyAsync(() -> riskyOperation())
    .exceptionally(e -> {
        System.out.println("错了：" + e.getMessage());
        return fallbackValue;
    });
```

## 4. JVM

### 4.1 JVM 内存区域划分

#### 正式版

JVM 运行时数据区分线程私有和线程共享。

**线程私有**：程序计数器（记录字节码执行位置）、虚拟机栈（每个方法对应一个栈帧，栈帧包含局部变量表/操作数栈/动态链接/方法出口）、本地方法栈（native 方法）。

**线程共享**：堆（存对象实例，GC 主要区域，分新生代 Eden/S0/S1 和老年代）、方法区（JDK 8 后为元空间 Metaspace，存类信息/常量/静态变量，使用本地内存）。

---

#### 详细版

**堆的分代：**
```
堆
├── 新生代（Young Generation）
│   ├── Eden（伊甸园，新对象分配的地方）
│   ├── S0（Survivor 0，存活区）
│   └── S1（Survivor 1，存活区）
│   默认占比：Eden : S0 : S1 = 8 : 1 : 1
│
└── 老年代（Old Generation，长期存活的对象）
```

**对象分配流程（面试重点）：**
```
new Object()
  ├── 1. 栈上分配（小对象 + 无逃逸）→ 直接栈上，GC 都不需要
  │    失败 ↓
  ├── 2. Eden 区分配
  │    Eden 不够 → Minor GC
  │    GC 后存活 → 移入 S0/S1
  │    年龄 +1
  │    年龄 > 15（默认）→ 移入老年代
  │    失败 ↓
  └── 3. 老年代分配
       老年代不够 → Full GC
```

**虚拟机栈：**
```
每个方法调用创建一个栈帧
栈帧包含：
├── 局部变量表：方法参数 + 局部变量（引用/基本类型）
├── 操作数栈：字节码指令操作的地方
├── 动态链接：指向运行时常量池的引用
└── 方法出口：方法返回的位置

递归太深 → 栈帧过多 → StackOverflowError
```

**JDK 8 为什么用元空间替代永久代？**
- 永久代是堆的一部分，有大小限制（`-XX:MaxPermSize`），容易 OOM
- 元空间使用本地内存（系统内存），默认不限制，避免永久代的 OOM
- 字符串常量池从永久代移到了堆中

---

### 4.2 GC 垃圾回收算法

#### 正式版

判断垃圾用**可达性分析算法**（GC Roots 出发，不可达即垃圾）。垃圾回收算法：**标记-清除**（有碎片）、**标记-复制**（新生代用，无碎片，浪费 10% 空间）、**标记-整理**（老年代用，无碎片，需移动对象）。GC Roots：栈引用、静态变量、常量引用、JNI 引用。

---

#### 详细版

**标记-复制（新生代）：**
```
GC 前：[活][活][垃][垃][活]   | 空闲区域
       └──────┬───────────────┘
              ↓ 把存活对象复制到空闲区域
       [活][活][活]          | 原来是垃圾的不管了
       连续排列，无碎片！
```

为什么新生代用复制算法：新生代对象 90% 以上活不过第一次 GC，只需复制少量存活对象，效率高。

**标记-整理（老年代）：**
```
GC 前：[活][垃][活][垃][活]
              ↓
       把存活对象往一端移动
GC 后：[活][活][活] ← 无碎片
```

为什么老年代用标记-整理：老年代存活率高，复制算法要复制大量对象，不合适。

**常用的垃圾收集器：**

| 收集器 | 适用区域 | 算法 | 特点 |
|--------|---------|------|------|
| Serial | 新生代 | 复制 | 单线程，暂停所有用户线程（STW） |
| ParNew | 新生代 | 复制 | Serial 的多线程版本 |
| Parallel Scavenge | 新生代 | 复制 | 关注吞吐量 |
| CMS | 老年代 | 标记-清除 | 并发，低延迟，有碎片 |
| G1 | 新生代+老年代 | 标记-整理+复制 | 区域化，可预测停顿，JDK 9+ 默认 |

**GC 触发的时机：**
- **Minor GC**：Eden 区空间不足。触发频繁，速度快
- **Full GC**：老年代空间不足。触发少，速度慢（STW 时间长）

### 4.3 类加载机制和双亲委派

#### 正式版

类加载过程：**加载**（读 class 文件进内存）→**连接**（验证格式、准备分配静态变量默认值、解析符号引用→直接引用）→**初始化**（执行 `<clinit>` 方法，给静态变量赋初值，执行静态代码块）。双亲委派模型：加载类时先委托父加载器，父加载不了才自己加载，保证核心类不被篡改。三层类加载器：Bootstrap（rt.jar）、Extension（ext 目录）、Application（classpath）。

---

#### 详细版

**类加载过程（三步）：**
```java
// 1. 加载
//    ClassLoader 读取 .class 文件字节流 → 方法区存类信息 → 堆中生成 Class 对象

// 2. 连接
//    验证：检查 class 文件格式（魔数 CAFEBABY）、字节码安全性
//    准备：为静态变量分配内存并设置默认值（int=0, Object=null）
//    解析：将常量池中的符号引用（如 CONSTANT_Class_info）替换为直接引用（内存地址）

// 3. 初始化
//    执行 <clinit> 方法（收集所有静态变量赋值和静态代码块，按代码顺序执行）
//    执行 main 方法（如果有）
```

**例子：`static int a = 10;` 的完整历程**
```
准备阶段：a = 0（默认值）
初始化阶段：a = 10（赋初值）
```

**双亲委派工作流程：**
```
ApplicationClassLoader 收到类加载请求
  └── 委托给 ExtensionClassLoader
      └── 委托给 BootstrapClassLoader
          ├── 能加载（如 java.lang.String）→ 加载并返回
          └── 不能加载 → 告诉子加载器
      └── ExtensionClassLoader 尝试加载
          ├── 能加载 → 加载并返回
          └── 不能加载 → 告诉子加载器
  └── ApplicationClassLoader 加载 classpath 下的类
```

**为什么要打破双亲委派？**

Tomcat 需要为每个 Web 应用加载不同的类。如果两个 Web 应用有不同版本的 Spring，父 ClassLoader 加载了一个，另一个就没法用了。Tomcat 的 WebAppClassLoader 打破双亲委派：先自己加载，加载不了才委托给父加载器。

---

### 4.4 对象创建过程

#### 正式版

JVM 创建一个对象（`new Object()`）的完整过程：**类加载检查**（检查能否在常量池定位到类的符号引用，类是否已加载）→**分配内存**（指针碰撞或空闲列表，取决于 GC 是否规整）→**初始化零值**（所有字段设为默认值）→**设置对象头**（Mark Word + 类指针）→**执行 `<init>` 方法**（执行构造方法，按代码顺序给字段赋值）。

---

#### 详细版

```java
User user = new User("张三", 18);
```

编译成字节码后，实际做了 5 步：

```
第 1 步：类加载检查
  JVM 检查常量池中能否定位到 User 的符号引用
  如果 User 类还没加载 → 先执行类加载过程（加载→连接→初始化）

第 2 步：分配内存
  从堆中划出一块内存给新对象
  GC 是标记-整理（如 G1）→ 指针碰撞（一个指针往后移）
  GC 是标记-清除（如 CMS）→ 空闲列表（维护一个空闲列表）
  多线程并发分配 → CAS 失败重试 或 TLAB（线程本地分配缓冲区）

第 3 步：初始化零值
  所有字段设为默认值：int=0, boolean=false, Object=null
  这保证了对象的字段在构造方法执行前就已经有初始值了

第 4 步：设置对象头
  对象头 Mark Word：锁状态（无锁）、GC 分代年龄（0）、identity hashcode（0）
  对象头类指针：指向方法区的 User 类元数据

第 5 步：执行 <init> 方法
  执行构造方法 User(String name, int age)
  按代码顺序：父类构造器 → 成员变量赋值 → 构造方法体
  this.name = name; this.age = age;
```

### 4.5 强引用 / 软引用 / 弱引用 / 虚引用

#### 正式版

**强引用**（`new Object()`）：只要引用还在，GC 永不回收。**软引用**（`SoftReference`）：内存不足时回收，适合做缓存。**弱引用**（`WeakReference`）：下次 GC 时立即回收（不管内存够不够），ThreadLocalMap 的 key 就是弱引用。**虚引用**（`PhantomReference`）：最弱，get 永远返回 null，仅用于跟踪对象被 GC 回收的消息（监控回收时机）。

---

#### 详细版

```java
// 强引用（最常见）
User user = new User();  // user 是强引用
user = null;             // 断开引用后 GC 才回收

// 软引用（内存不足才回收）
SoftReference<User> softRef = new SoftReference<>(new User());
User user = softRef.get();  // 内存充足时返回对象，不足时返回 null
// 适合做缓存：图片缓存、大对象缓存

// 弱引用（下次 GC 就回收）
WeakReference<User> weakRef = new WeakReference<>(new User());
User user = weakRef.get();  // GC 后返回 null
// ThreadLocalMap 的 Entry 继承 WeakReference，key 是弱引用
// 这样 ThreadLocal 对象外部引用断开后，GC 可以回收 key

// 虚引用（最弱，get 永远 null）
ReferenceQueue<User> queue = new ReferenceQueue<>();
PhantomReference<User> phantomRef = new PhantomReference<>(new User(), queue);
User user = phantomRef.get();  // 永远返回 null
// 虚引用主要用来监控对象何时被回收：对象被回收时 ReferenceQueue 收到通知
// NIO 中 DirectByteBuffer 的回收就用到了虚引用
```

## 5. Spring / Spring Boot

### 5.1 IoC 和 DI

#### 正式版

IoC（控制反转）：对象的创建和依赖管理从程序员交给 Spring 容器。DI（依赖注入）：容器在创建对象时自动把依赖传进去。推荐**构造器注入**（final 字段、不可变、测试友好），避免字段注入 `@Autowired`。三种注入方式对比：构造器注入（强制依赖，推荐）、Setter 注入（可选依赖）、字段注入（不推荐，无法用 final，不方便测试）。

---

### 5.2 Bean 生命周期

#### 正式版

**实例化**（反射创建对象）→**属性赋值**（注入依赖）→**Aware 回调**（BeanNameAware 等）→**BeanPostProcessor.before**→**初始化**（@PostConstruct / afterPropertiesSet / initMethod）→**BeanPostProcessor.after**（AOP 代理在这里生成）→**使用中**→**销毁**（@PreDestroy / destroy）。

---

#### 详细版

```java
@Component
public class MyBean implements InitializingBean, DisposableBean {
    
    public MyBean() { System.out.println("1. 实例化"); }

    @Autowired private OtherBean other;
    // 2. 属性赋值
    
    @Override
    public void setBeanName(String name) { System.out.println("3. BeanNameAware"); }

    @PostConstruct
    public void init() { System.out.println("5. @PostConstruct"); }

    @Override
    public void afterPropertiesSet() { System.out.println("6. afterPropertiesSet"); }

    @Bean(initMethod = "customInit")
    public void customInit() { System.out.println("7. initMethod"); }

    // BeanPostProcessor.afterInitialization 在这里（可选）生成 AOP 代理
    // 如果类有 @Transactional，这里返回的是 CGLIB 代理对象

    @PreDestroy
    public void preDestroy() { System.out.println("8. @PreDestroy"); }

    @Override
    public void destroy() { System.out.println("9. DisposableBean.destroy"); }
}
```

**第 7 步（BeanPostProcessor.afterInitialization）为什么重要：**

这是 AOP 代理生成的地方。如果一个 Bean 有 `@Transactional` 或 `@Cacheable` 注解，Spring 在这里创建一个代理对象替换原来的 Bean。所以其他地方注入的不是原始 Bean，而是代理对象。这也是 `@Transactional` 在同一个类内部互相调用时不生效的原因——内部调用是 `this.method()`，走的是原始对象，不是代理对象。

### 5.3 Spring Boot 自动配置原理

#### 正式版

`@SpringBootApplication` → `@EnableAutoConfiguration` → `AutoConfigurationImportSelector` → 读取 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件中的自动配置类列表。每个自动配置类用 `@ConditionalOnClass`（classpath 有类才生效）、`@ConditionalOnMissingBean`（用户没自定义才生效）、`@ConditionalOnProperty`（配置存在才生效）等条件注解控制是否生效。

---

#### 详细版

```java
// 简化版的 RedisAutoConfiguration
@AutoConfiguration
@ConditionalOnClass(RedisOperations.class)  // 有 Redis 依赖才生效
@EnableConfigurationProperties(RedisProperties.class)  // application.yml 中 spring.data.redis.*
public class RedisAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean(name = "redisTemplate")
    // 用户没有自定义 RedisTemplate 时才创建默认的
    public RedisTemplate<Object, Object> redisTemplate(RedisConnectionFactory factory) {
        RedisTemplate<Object, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(factory);
        return template;
    }
}
```

```
-----------------
RabbitAutoConfiguration:
  - @ConditionalOnClass did not find required class '...'
```

### 5.4 @Transactional 原理和失效场景

#### 正式版

`@Transactional` 基于 AOP 代理实现。方法执行前开启事务，执行后提交事务，异常时回滚。**失效场景**（面试高频）：**1. 同一个类内方法调用**（`this.method()` 走原始对象，不走代理，事务注解失效）。**2. 方法不是 public**（@Transactional 只对 public 方法生效）。**3. 捕获了异常没抛**（事务默认只回滚 RuntimeException，如果 catch 了没抛不会回滚）。**4. 数据库引擎不支持事务**（MyISAM）。**5. 事务传播行为配置不当**。

---

#### 详细版

**失效场景 1：内部调用（最常见）**
```java
@Service
public class UserService {
    public void save() {
        doSave();  // this.doSave()，走原始对象，事务不生效！
    }
    @Transactional
    public void doSave() {
        // 即使这里抛异常，也不会回滚
    }
}
```
解决：把事务方法提取到另一个 Service 中注入调用，或注入自己（`@Autowired UserService self`，然后 `self.doSave()`）

**失效场景 2：异常被吃了**
```java
@Transactional
public void save() {
    try {
        // 操作数据库
        throw new RuntimeException();
    } catch (Exception e) {
        // 捕获了异常没抛 → 事务认为执行成功 → 提交了！
    }
}
```
解决：`catch` 中抛异常或 `@Transactional(rollbackFor = Exception.class)` 并 `throw e`

**失效场景 3：异常类型不对**
```java
@Transactional
public void save() throws Exception {
    throw new Exception();  // 不是 RuntimeException，不会回滚！
}
// @Transactional 默认只回滚 RuntimeException 和 Error
// checked exception（如 Exception）不回滚
```
解决：`@Transactional(rollbackFor = Exception.class)`

### 5.5 Spring 循环依赖（三级缓存）

#### 正式版

Spring 通过**三级缓存**解决单例 setter 注入的循环依赖（构造器注入无法解决）。一级缓存 `singletonObjects`（成品 Bean）、二级缓存 `earlySingletonObjects`（半成品 Bean，已实例化未初始化）、三级缓存 `singletonFactories`（工厂，提前暴露 Bean 引用）。Spring 在实例化 Bean 后就把工厂放入三级缓存，其他 Bean 依赖时通过工厂提前暴露一个半成品引用，等所有依赖注入完成再初始化。

---

#### 详细版

**场景：A 依赖 B，B 依赖 A**

```
A 创建过程：
  1. 实例化 A（调用构造器，对象创建了，但属性还是 null）
  2. 把 A 的工厂放入三级缓存（singletonFactories）
  3. 填充属性：需要 B
      ↓
    发现 B 还没创建，先创建 B：
      3.1 实例化 B
      3.2 把 B 的工厂放入三级缓存
      3.3 填充属性：需要 A
          → 从三级缓存找到 A 的工厂（A 已经实例化了）
          → 通过工厂获取 A 的半成品引用（early exposure）
          → 把 A 放入二级缓存（earlySingletonObjects）
          → 注入到 B 中
      3.4 B 初始化完成 → B 放入一级缓存（成品）
      ↑
  4. A 拿到 B 的引用（B 已经是成品了）
  5. A 初始化完成 → A 放入一级缓存
```

**三级缓存分别对应什么：**
```java
// 一级缓存：成品 Bean（已完全初始化）
Map<String, Object> singletonObjects;

// 二级缓存：半成品 Bean（已实例化，但还没完成属性赋值和初始化）
Map<String, Object> earlySingletonObjects;

// 三级缓存：生产半成品 Bean 的工厂（提前暴露引用）
Map<String, ObjectFactory<?>> singletonFactories;
```

**为什么三级缓存是工厂而不是直接存半成品？**

三级缓存存的是 `ObjectFactory`，可以在获取半成品的时候执行一些处理逻辑（比如 AOP 代理的生成）。如果不需要 AOP，工厂返回的就是原始半成品。这样可以做到"**懒初始化**"——只有真正发生循环依赖时才去生成半成品。

**什么情况下解决不了？**

构造器注入的循环依赖无法解决：
```java
// A 的构造器需要 B，B 的构造器需要 A
// Spring 在实例化阶段就卡住了——都等着对方构造完成
public A(B b) { }
public B(A a) { }
// 解决方案：改用 setter 注入或 @Lazy
```

### 5.6 Spring AOP 实现原理

#### 正式版

Spring AOP 基于**动态代理**实现。目标类实现接口 → **JDK 动态代理**（Proxy + InvocationHandler，生成实现了相同接口的代理对象）。目标类没实现接口 → **CGLIB 代理**（生成目标类的子类）。代理对象在目标方法执行前后插入切面逻辑。Spring AOP 只支持**方法级别**的切面（不像 AspectJ 支持字段/构造器级别）。

---

#### 详细版

**JDK 动态代理：**
```java
// 目标对象实现了接口
public interface UserService { void save(); }
public class UserServiceImpl implements UserService { public void save() { ... } }

// JDK 动态代理
UserService proxy = (UserService) Proxy.newProxyInstance(
    classLoader,
    new Class[]{UserService.class},
    (proxy, method, args) -> {
        System.out.println("前置增强");
        Object result = method.invoke(target, args);  // 调用目标方法
        System.out.println("后置增强");
        return result;
    }
);
proxy.save();  // 调用的是代理对象的方法
```

**CGLIB 代理：**
```java
// 目标对象没有实现接口（如项目中的 Controller）
// CGLIB 动态生成子类
public class TaskControllerProxy extends TaskController {
    @Override
    public Result createTask(...) {
        System.out.println("前置增强");
        Object result = super.createTask(...);  // 调用父类方法
        System.out.println("后置增强");
        return result;
    }
}
```

**两种代理对比：**

| | JDK 动态代理 | CGLIB 代理 |
|--|------------|-----------|
| 条件 | 目标必须有接口 | 目标不要求接口 |
| 如何生成 | 反射 API `Proxy.newProxyInstance` | 字节码框架 ASM 生成子类字节码 |
| 性能（创建） | 更高 | 较低（生成字节码） |
| 性能（调用） | 调用反射（稍慢） | 直接调用父类（快） |
| 限制 | 只能代理接口方法 | final 方法和类无法代理 |

---

### 5.7 事务传播行为

#### 正式版

事务传播行为定义了**两个有事务的方法互相调用时，事务如何传播**。7 种传播行为：**REQUIRED**（默认，支持当前事务，没有就新建）、**REQUIRES_NEW**（总是新建事务，挂起当前事务）、**SUPPORTS**（有就用，没有就不用）、**NOT_SUPPORTED**（以非事务方式执行）、**MANDATORY**（必须有事务）、**NEVER**（必须没有事务）、**NESTED**（嵌套事务，有独立回滚点）。

---

#### 详细版

```java
@Service
public class OuterService {
    @Autowired
    private InnerService innerService;

    @Transactional
    public void outer() {
        // 当前事务 T1
        innerService.inner();  // 调用内层方法
        // 如果这里抛异常 → T1 回滚，inner 的修改也会回滚
    }
}

@Service
public class InnerService {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void inner() {
        // 新事务 T2（T1 被挂起）
        // inner 提交时 T2 提交，outer 回滚时不影响 T2
    }
}
```

| 传播行为 | 外层有事务 | 外层无事务 |
|---------|-----------|-----------|
| REQUIRED（默认） | 加入外层 | 新建 |
| REQUIRES_NEW | 挂起外层，新建独立事务 | 新建 |
| SUPPORTS | 加入外层 | 非事务执行 |
| NOT_SUPPORTED | 挂起外层，非事务执行 | 非事务执行 |
| MANDATORY | 加入外层 | 抛异常 |
| NEVER | 抛异常 | 非事务执行 |
| NESTED | 新建嵌套事务（有回滚点） | 新建 |

## 6. MySQL

### 6.1 B+ 树索引结构

#### 正式版

InnoDB 使用 B+ 树作为索引结构。非叶子节点只存键和指针（不存数据），一个 16KB 的页可以存约 1170 个键。叶子节点存数据（聚簇索引存整行，二级索引存主键值），叶子节点间通过双向链表连接。3 层 B+ 树可存约 16 亿条数据，查询只需 2-3 次磁盘 IO。相比于 B 树：B+ 树非叶子节点不存数据→树更矮→IO 更少→查询更稳定。

---

#### 详细版

**B+ 树和 B 树的核心区别：**
```
B 树：非叶子节点也存数据
  ┌────────┐
  │ key+数据│ → 每个节点能存的 key 少
  └────────┘      树的高度更高

B+ 树：非叶子节点只存 key
  ┌──────┐
  │ key  │ → 每个节点能存很多 key（约 1170 个）
  └──────┘      树的高度更低（2-4 层）
  ┌──────────┐
  │ key+数据 │ → 所有数据在叶子层
  └──────────┘    且叶子层有序链表连接
```

**计算三层 B+ 树能存多少数据：**
```
InnoDB 一页 = 16KB
bigint 主键 = 8 字节 + 指针 = 6 字节 ≈ 14 字节
一个节点存放的 key 数 ≈ 16KB / 14B ≈ 1170

第 1 层（根）：1170 个指针
第 2 层：1170 × 1170 = 1,368,900 个指针
第 3 层（叶子）：1,368,900 × 1170 ≈ 16 亿条数据（叶子层存数据）

所以 3 层 B+ 树可以管理 16 亿条数据，查询只需要 3 次磁盘 IO
```

**聚簇索引 vs 二级索引：**
```sql
CREATE TABLE user (
    id INT PRIMARY KEY,    -- 聚簇索引，叶子存整行数据
    name VARCHAR(50),
    age INT,
    INDEX idx_name(name)   -- 二级索引，叶子存 id
);

-- 回表查询：
SELECT * FROM user WHERE name = '张三';
-- 1. 去 idx_name 的 B+ 树找 "张三" → 找到 id = 1
-- 2. 再回主键索引的 B+ 树找 id = 1 → 找到整行数据
-- 两次查询 = 回表

-- 覆盖索引（不回表）：
SELECT id, name FROM user WHERE name = '张三';
-- idx_name 的叶子节点已经有 id 和 name 了，不需要回表
```

### 6.2 SQL 优化和 EXPLAIN

#### 正式版

用 `EXPLAIN` 分析 SQL，关注这几个关键列：**type**（const > ref > range > index > ALL，至少到 range）、**rows**（扫描行数，越小越好）、**Extra**（`Using index` 覆盖索引最好；`Using filesort` 和 `Using temporary` 需要优化，说明没用到索引排序/分组）。常见优化：对 WHERE/ORDER BY/GROUP BY 涉及的列建索引；避免 SELECT *；避免在索引列上使用函数；用 EXISTS 替代 IN（子查询太大时）。

---

#### 详细版

```sql
EXPLAIN SELECT * FROM user WHERE name = '张三';
```

**输出各列解释：**

| 列 | 示例值 | 含义 |
|-----|-------|------|
| id | 1 | 查询序号 |
| select_type | SIMPLE | 查询类型（SIMPLE/PRIMARY/SUBQUERY/DERIVED） |
| **type** | **ref** | **访问类型（性能从好到差）** |
| possible_keys | idx_name | 可能用到的索引 |
| **key** | **idx_name** | **实际用到的索引** |
| key_len | 202 | 索引使用的字节数 |
| **rows** | **1** | **预估扫描行数（越小越好）** |
| filtered | 100 | 过滤后剩余百分比 |
| **Extra** | **Using index** | **额外信息** |

**type 性能排序：**
```
null（不访问表） > const（主键/唯一） > eq_ref（JOIN 主键） 
> ref（普通索引等值） > range（索引范围） > index（全索引扫描）
> ALL（全表扫描，要优化！）
```

**Extra 常见值及意义：**

| Extra | 含义 | 优化建议 |
|-------|------|---------|
| Using index | **覆盖索引**，没回表 | ✅ 最好 |
| Using where | 回表后在服务器层过滤 | 正常，看 rows 大小 |
| Using index condition | **索引下推**（ICP） | ✅ 好 |
| **Using filesort** | 文件排序（不是索引排序） | ❌ 建联合索引优化 |
| **Using temporary** | 用了临时表（GROUP BY 没索引） | ❌ 建联合索引优化 |
| Using join buffer | JOIN 没有用索引 | ❌ 为 JOIN 列建索引 |

**常见优化案例：**
```sql
-- 1. 索引列上使用函数 → 索引失效
-- 慢
SELECT * FROM order WHERE YEAR(create_time) = 2026;
-- 快
SELECT * FROM order WHERE create_time >= '2026-01-01' AND create_time < '2027-01-01';

-- 2. LIKE 前导通配符 → 索引失效
-- 慢
SELECT * FROM user WHERE name LIKE '%张%';
-- 非前导通配符可以用索引
SELECT * FROM user WHERE name LIKE '张%';

-- 3. 大分页
-- 慢（要扫描 100010 行，丢弃前 100000）
SELECT * FROM order ORDER BY id LIMIT 100000, 10;
-- 快（游标分页，直接定位）
SELECT * FROM order WHERE id > 100000 ORDER BY id LIMIT 10;

-- 4. JOIN 的列建立索引
SELECT * FROM order o LEFT JOIN user u ON o.user_id = u.id;
-- user.id 有主键索引，但 order.user_id 要有索引
```

---

### 6.3 事务 ACID 和隔离级别

#### 正式版

**原子性**：全成功或全回滚（undo log 实现）。**一致性**：事务前后数据一致（由 AID 共同保证）。**隔离性**：并发事务不互相干扰（MVCC + 锁实现）。**持久性**：事务提交后数据不丢（redo log 实现）。

隔离级别从低到高：**RU**（读未提交，有脏读）< **RC**（读已提交，Oracle 默认，有不可重复读）< **RR**（可重复读，MySQL 默认，有幻读，InnoDB 通过间隙锁解决）< **Serializable**（串行化，性能最差）。脏读：读到未提交的数据。不可重复读：两次读同一行结果不同。幻读：两次范围查询结果行数不同。

---

#### 详细版

**ACID 的实现：**
```
A 原子性：undo log
  事务执行时记录修改前的值。
  回滚时用 undo log 恢复。

C 一致性：AID 共同保证
  转账前后总金额不变。

I 隔离性：MVCC + 锁
  读不加锁，写加锁，读写不冲突。

D 持久性：redo log
  提交时先写 redo log（顺序写磁盘），再写数据页（随机写）。
  如果断电，重启时用 redo log 恢复已提交的事务。
  redo log 是物理日志（"在哪个页的哪个偏移量写了什么数据"）。
```

**三种并发问题示例：**

脏读（Dirty Read）：
```
事务 A                         事务 B
开启事务                       开启事务
                                UPDATE account SET balance=200 WHERE id=1
SELECT balance → 200（脏读！
B 还没提交，数据可能回滚）
                                 ROLLBACK（回滚了，balance 其实是 100）
```

不可重复读（Non-Repeatable Read）：
```
事务 A                         事务 B
开启事务
SELECT balance → 100
                                UPDATE account SET balance=200 WHERE id=1
                                COMMIT
SELECT balance → 200（同一个事务
两次读到的值不一样！）
```

幻读（Phantom Read）：
```
事务 A                         事务 B
开启事务
SELECT * FROM account WHERE balance>0 → 1 条
                                INSERT INTO account VALUES (2, 300)
                                COMMIT
SELECT * FROM account WHERE balance>0 → 2 条（多了 1 条，像幻觉）
```

**MySQL InnoDB 默认 RR 解决幻读：**
```sql
-- RR 级别下，InnoDB 用间隙锁（Gap Lock）解决幻读
-- 间隙锁锁住的是索引之间的间隙，不让新的记录插入到这个间隙中
SELECT * FROM account WHERE balance > 100 FOR UPDATE;
-- 不仅锁住了已有行，还锁住了 balance > 100 这个范围（间隙）
-- 其他事务不能在这个范围内插入新行
```

### 6.4 MVCC 原理

#### 正式版

MVCC（多版本并发控制）让 InnoDB 实现"读不加锁，读写不冲突"。每行数据有多个版本（undo log 版本链），每个版本带事务 ID。事务查询时根据**一致性视图（Read View）**确定可见性。Read View 包含 `up_limit_id`（最小活跃事务）、`low_limit_id`（最大事务+1）、`m_ids`（活跃事务列表）等。RR 级别在事务开始时创建 Read View 并复用。RC 级别每次查询都创建新 Read View（导致不可重复读）。

---

#### 详细版

**核心机制：**

每条记录有两个隐藏字段：
- `DB_TRX_ID`：最后修改这条记录的事务 ID
- `DB_ROLL_PTR`：指向 undo log 的指针

```
版本链（通过 DB_ROLL_PTR 串联）：
(id=1, balance=100, trx_id=10) ← 最早的版本
    ↑
(id=1, balance=200, trx_id=15) ← 中间版本
    ↑
(id=1, balance=300, trx_id=20) ← 当前最新版本（表中数据）
```

**Read View 的可见性判断：**
```
假设 Read View 生成时：up_limit_id=12, low_limit_id=21, m_ids=[12,15,18]

对于版本链中某个版本（trx_id=X）：
    ├── X == 当前事务 ID → 可见（自己改的自己当然能看到）
    ├── X < 12（up_limit_id）→ 可见（已提交的老事务）
    ├── X ≥ 21（low_limit_id）→ 不可见（还没开始的事务）
    ├── X ∈ [12,15,18]（m_ids）→ 不可见（还没提交）
    └── X ∉ m_ids → 可见（已提交）
```

**RR 和 RC 的区别：**
- **RR**：事务开始时创建 Read View，整个事务用同一个 → 不管别的事务怎么改，看到的数据不变 → 解决了不可重复读
- **RC**：每次 SELECT 都创建新 Read View → 别的事务提交了，新 Read View 就看不到了 → 拿到新数据 → 不可重复读

---

### 6.5 最左前缀法则

#### 正式版

联合索引遵循最左前缀法则：查询条件从索引的最左列开始，不跳过中间的列。如联合索引 `(a, b, c)`，`WHERE a=? AND b=?` 可以用索引，`WHERE b=? AND c=?`（没用到 a）不能。MySQL 会一直向右匹配直到遇到范围查询（`>`、`<`、`BETWEEN`、`LIKE`）就停止匹配。

---

#### 详细版

```sql
-- 联合索引 (a, b, c)

-- ✅ 用到索引
WHERE a = 1                      -- 用到 a
WHERE a = 1 AND b = 2            -- 用到 a, b
WHERE a = 1 AND b = 2 AND c = 3  -- 用到 a, b, c
WHERE a = 1 AND c = 3            -- 用到 a（c 用不到，因为跳过了 b）
WHERE a = 1 AND b > 2 AND c = 3  -- 用到 a, b（c 用不到，b 是范围查询后停止）

-- ❌ 用不到索引
WHERE b = 2                      -- 没用到 a
WHERE c = 3                      -- 没用到 a
WHERE b = 2 AND c = 3            -- 没用到 a
```

**为什么最左前缀？**

B+ 树的联合索引排序规则：先按 a 排，a 相同再按 b 排，b 相同再按 c 排。所以 `(b, c)` 在没有 a 的情况下，索引中的顺序是杂乱无章的，没法快速定位。

**常见误区：**
```sql
-- WHERE 条件的顺序不影响（查询优化器会重排）
WHERE b = 2 AND a = 1   -- 和 WHERE a = 1 AND b = 2 效果一样，都能用到 a, b

-- 不等于不能用索引
WHERE a != 1            -- 不能用索引

-- OR 可能导致索引失效
WHERE a = 1 OR b = 2    -- 可能全表扫描（取决于优化器）
```

---

### 6.6 索引下推

#### 正式版

索引下推（ICP，Index Condition Pushdown）是 MySQL 5.6 引入的优化。没有 ICP 时，存储引擎根据索引条件找到记录后回表，再把完整行返回给 Server 层过滤。有 ICP 时，存储引擎可以在索引层面直接过滤不符合条件的记录，减少回表次数。ICP 在 `Extra` 列显示为 `Using index condition`。

---

#### 详细版

```sql
-- 联合索引 (name, age)
SELECT * FROM user WHERE name LIKE '张%' AND age = 18;
```

**没有 ICP 时：**
```
1. 存储引擎用 idx_name_age 找到 name LIKE '张%' 的所有主键
   → 找到 "张三,16"、"张四,18"、"张五,20" 三个主键
2. 回表三次，把完整行数据返回给 Server 层
3. Server 层再过滤 age = 18
4. 结果：只有张四符合
```

**有 ICP 时：**
```
1. 存储引擎用 idx_name_age 找到 name LIKE '张%' 的记录
   → 在索引中就检查 age = 18（索引中包含 age 字段！）
   → 只返回 "张四,18" 一个主键
2. 回表一次，返回完整行
3. 省了两次回表！
```

**ICP 的前提条件：**
- 适用于二级索引（聚簇索引不需要，因为叶子节点直接存整行）
- 索引中包含查询涉及的字段
- 存储引擎层下推（InnoDB 支持）

## 7. Redis

### 7.1 5 种基本数据结构

#### 正式版

**String**：最基础，可存 JSON/数字，支持 INCR/DECR 原子操作。**List**：双向链表，LPUSH/RPOP 可做消息队列。**Set**：无序集合，去重、交并差运算。**ZSet**：有序集合，按 score 排序，适合排行榜。**Hash**：类似 Java HashMap，适合存对象。数据结构的底层编码有压缩列表、跳表、整数集合等（根据元素数量和大小自动切换）。

---

#### 详细版

```bash
# String — 缓存、计数器、分布式锁
SET user:1 '{"name":"张三"}' EX 3600  # 带过期时间
SETNX lock 1                          # 不存在才设置（分布式锁的基础）
INCR counter                          # 原子自增（项目限流用的）
GET user:1

# List — 消息队列、最新消息列表
LPUSH messages "msg1"                 # 从左边推入
RPOP messages                         # 从右边弹出
BRPOP messages 0                      # 阻塞弹出（没有消息时等待）

# Set — 在线用户、共同好友
SADD online:users "user:1"            # 添加
SREM online:users "user:1"            # 移除
SISMEMBER online:users "user:1"       # 判断是否存在
SINTER set1 set2                      # 交集（共同好友）

# ZSet — 排行榜、延迟队列
ZADD ranking 100 "user:1"             # 添加/更新，score=100
ZINCRBY ranking 50 "user:1"           # 加分
ZREVRANGE ranking 0 9 WITHSCORES      # 前 10 名
ZREM ranking "user:1"                 # 移除

# Hash — 对象缓存
HSET user:1 name "张三" age "18"      # 设置多个字段
HGET user:1 name                      # 获取一个字段
HGETALL user:1                        # 获取所有
HINCRBY user:1 points 10              # 字段自增
```

### 7.2 过期策略和内存淘汰

#### 正式版

**过期策略**：**定期删除**（每 100ms 随机抽取部分过期 key 检查删除） + **惰性删除**（访问 key 时检查过期了就删）。组合保证了不过度消耗 CPU 也不会漏删。**内存淘汰策略**（内存满了时）：`allkeys-lru`（从所有 key 中淘汰最近最少使用的，推荐）、`volatile-lru`（从设了过期时间的 key 中淘汰 LRU）、`allkeys-random`、`volatile-ttl`（淘汰即将过期的）、`noeviction`（不淘汰，写时报错，默认）。

---

### 7.3 Redis 持久化（RDB / AOF）

#### 正式版

**RDB**（快照）：按时间间隔生成全量快照（`SAVE` 阻塞、`BGSAVE` 非阻塞，fork 子进程写文件）。适合备份和灾难恢复，但可能丢失最后一次快照后的数据。**AOF**（追加文件）：记录每个写命令，文件追加方式。恢复时重放命令。AOF 有三种刷盘策略：`always`（每条命令都刷，最安全但最慢）、`everysec`（每秒刷一次，推荐）、`no`（由 OS 决定）。AOF 文件会越来越大，需定期重写（`BGREWRITEAOF`）。**混合持久化**（Redis 4.0+）：RDB 做全量快照 + AOF 做增量日志，兼顾启动速度和数据安全。

---

### 7.4 缓存穿透 / 击穿 / 雪崩

#### 正式版

**缓存穿透**：查不存在的数据，缓存和数据库都没有，请求直接打到数据库。解决：缓存空值（短过期时间）或布隆过滤器。

**缓存击穿**：热点 key 过期瞬间大量并发请求。解决：分布式锁 + 双重检查（第一个线程拿到锁回去查数据库回写缓存，后面的查缓存即可），或热点 key 永不过期。

**缓存雪崩**：大量 key 同时过期或 Redis 挂了。解决：过期时间加随机值分散；Redis 高可用（主从 + 哨兵/Cluster）；多级缓存（本地缓存 Caffeine + Redis）。

---

### 7.5 Redis 分布式锁

#### 正式版

最简实现：`SET key value NX PX 30000`（不存在才设置，30 秒自动过期）。但手写有很多坑：不可重入、没有自动续期（Watchdog）、可能锁误释放。生产环境推荐用 **Redisson**，它提供了可重入 RLock、Watchdog 自动续期、获取锁的原子性保证。在 Redisson 中，`lock.lock()` 默认 30 秒过期并有 Watchdog 续期，`tryLock(wait, lease, unit)` 指定等待时间和租约时间（设了 leaseTime 就不启动 Watchdog）。

---

### 7.6 缓存一致性（双写模式）

#### 正式版

更新数据库 + 删除/更新缓存时可能出现数据不一致。推荐策略：**先更新数据库，再删除缓存**（Cache Aside Pattern）。为什么不是先删缓存再更新数据库？因为并发下 A 删了缓存，B 读到了旧数据并写回缓存，数据就会长期不一致。先更新数据库再删缓存，即使删缓存失败，也只是短暂的不一致（缓存过期后自动恢复）。最终方案：**延迟双删**（先删缓存 → 更新数据库 → 延迟 N 毫秒再删缓存），或通过消息队列保证删除缓存的可靠性。

## 8. 计算机网络

### 8.1 三次握手和四次挥手

#### 正式版

**三次握手**：客户端发 SYN → 服务端回 SYN+ACK → 客户端发 ACK。**四次挥手**：主动方发 FIN → 被动方回 ACK → 被动方发 FIN → 主动方回 ACK，主动方 TIME_WAIT 2MSL 后关闭。三次握手的核心是**防止失效的连接请求到达服务端**（网络延迟导致的过期 SYN）。TIME_WAIT 的目的是保证最后一个 ACK 能到达（如果丢了可以重发）和使过期数据包消失。

---

#### 详细版

**三次握手（建立连接）：**
```
TCP 是可靠的、面向连接的协议，需要先建立连接再传数据。

客户端（CLOSED）                   服务端（LISTEN）
  │                                     │
  │ 1. SYN=1, seq=x                      │  客户端发送 SYN，进入 SYN_SENT
  │ ─────────────────────────────────▶  │  服务端收到，知道客户端要连接
  │                                     │
  │ 2. SYN=1, ACK=1, seq=y, ack=x+1    │  服务端回 SYN+ACK，进入 SYN_RCVD
  │ ◀───────────────────────────────── │  客户端收到，知道服务端能收能发
  │                                     │
  │ 3. ACK=1, seq=x+1, ack=y+1          │  客户端发 ACK，进入 ESTABLISHED
  │ ─────────────────────────────────▶  │  服务端收到，知道客户端能收
  │                                     │  连接建立完成，可以传输数据了
```

为什么是三次不是两次？如果是两次，第一个 SYN 在网络中延迟了，客户端超时重发 SYN 并成功建立连接、传输数据、关闭连接。然后第一个延迟的 SYN 到了服务端，服务端以为是个新连接，创建连接等待数据——但客户端不会理它（没有第三次握手），服务端白白浪费资源。

**四次挥手（断开连接）：**
```
TCP 是全双工的，两个方向需要分别关闭。

主动方                           被动方
  │                                │
  │ 1. FIN=1, seq=u                │  主动方关闭发送，进入 FIN_WAIT_1
  │ ──────────────────────────▶    │  被动方收到，知道对方不发数据了
  │                                │
  │ 2. ACK=1, seq=v, ack=u+1      │  被动方回 ACK，进入 CLOSE_WAIT
  │ ◀──────────────────────────   │  主动方收到，进入 FIN_WAIT_2
  │                                │  （被动方还可以继续发送数据）
  │                                │
  │ 3. ... 被动方继续发剩余数据 ...   │
  │                                │
  │ 4. FIN=1, seq=w, ack=u+1      │  被动方发 FIN，进入 LAST_ACK
  │ ◀──────────────────────────   │  （被动方也没数据了）
  │                                │
  │ 5. ACK=1, seq=u+1, ack=w+1    │  主动方回 ACK，进入 TIME_WAIT
  │ ──────────────────────────▶    │  被动方收到，关闭连接
  │                                │  主动方等待 2MSL 后关闭
```

为什么 TIME_WAIT 要等 2MSL？
1. 保证最后一个 ACK 能被收到。如果 ACK 丢了，被动方会重发 FIN，主动方在 TIME_WAIT（等 2MSL）期间可以重新发 ACK
2. 让本次连接的所有数据包在网络中消失，不影响下次使用相同端口的新连接

### 8.2 HTTP 和 HTTPS

#### 正式版

HTTPS = HTTP + SSL/TLS，解决的问题：**窃听**（加密传输）、**篡改**（MAC 校验）、**冒充**（数字证书验证身份）。HTTPS 握手先 TCP 三次握手，再进行 TLS 握手：服务端返回证书 → 客户端验证证书 → 用证书公钥加密传输对称密钥（**非对称加密**保护对称密钥的传输）→ 后续数据传输用**对称加密**。HTTP 端口 80，HTTPS 端口 443。

---

### 8.3 输入 URL 到页面展示的完整过程

#### 正式版

1. **DNS 解析**：浏览器缓存 → 系统缓存 → 路由器缓存 → ISP DNS 服务器 → 根/顶级/权威 DNS 服务器
2. **TCP 连接**：三次握手建立连接（HTTPS 额外 TLS 握手）
3. **发送 HTTP 请求**：浏览器构造请求行/头/体，发送给服务端
4. **服务端处理**：反向代理（Nginx）→ Web 容器（Tomcat）→ Spring 框架 → Controller → Service → DB → 返回响应
5. **浏览器渲染**：解析 HTML 构建 DOM 树 → 解析 CSS 构建 CSSOM 树 → 合成渲染树 → 布局（Layout）→ 绘制（Paint）
6. **资源加载**：遇到 `<script>` 阻塞解析（可加 async/defer）→ 遇到 `<img>` 发 GET 请求加载

---

### 8.4 HTTP 1.0 / 1.1 / 2.0 / 3.0 的区别

#### 正式版

**HTTP 1.0**：每个请求新建 TCP 连接，用完关闭，效率低。**HTTP 1.1**：引入持久连接（Keep-Alive，TCP 复用）、管道化（Pipelining，但 HOL 阻塞）、分块传输编码。**HTTP 2.0**：多路复用（一个 TCP 连接并行多个请求，解决了 HOL 阻塞）、头部压缩（HPACK）、服务端推送（Server Push）。**HTTP 3.0**：基于 UDP 的 QUIC 协议，0-RTT 连接建立，解决了 TCP 队头阻塞问题。

---

### 8.5 Cookie、Session、Token 的区别

#### 正式版

**Cookie**：浏览器存储的小型数据（4KB），每次请求自动带同域的 Cookie，可设置过期时间。**Session**：服务端存储的用户状态信息，通过 Cookie 中的 Session ID 关联。分布式部署需要 Session 共享（Redis）。**Token**（JWT）：无状态认证，服务端不存储 Token，客户端持有包含用户信息的签名字符串。Token 存放在请求头而非 Cookie，天然免疫 CSRF。Session 适合传统服务端渲染，JWT 适合前后端分离/移动端。

---

### 8.6 HTTP 常见状态码

#### 正式版

**1xx**：信息，101 切换协议（WebSocket 升级）。**2xx**：成功，200 OK、201 Created（创建成功）、204 No Content（删除成功）。**3xx**：重定向，301 永久重定向、302 临时重定向、304 未修改（缓存）。**4xx**：客户端错误，400 参数错误、401 未认证、403 权限不足、404 不存在、405 方法不允许、**429 请求太频繁**（项目限流用的）、**422 参数校验失败**。**5xx**：服务端错误，500 服务器内部错误、502 网关错误、503 服务不可用、504 网关超时。

## 9. 操作系统

### 9.1 进程和线程的区别

#### 正式版

进程是资源分配的最小单位，线程是 CPU 调度的最小单位。进程有独立地址空间（一个崩了不影响其他），线程共享进程地址空间（一个崩了整个进程挂）。进程切换开销大（页表/TLB/缓存刷新），线程切换开销小（寄存器/PC）。多线程要处理同步问题。

---

### 9.2 死锁的四个必要条件

#### 正式版

**互斥**：资源一次只能被一个线程占用。**持有并等待**：线程持有资源且正在等待其他资源。**不可剥夺**：线程持有的资源不能被强制拿走。**循环等待**：线程 A 等 B 的资源，B 等 C 的资源，C 等 A 的资源，形成循环。破坏任意一个条件就能避免死锁。项目中通过**锁排序**（所有线程按相同顺序获取锁）破坏循环等待。

---

#### 详细版

```java
// 死锁示例
public class DeadlockDemo {
    private static final Object lockA = new Object();
    private static final Object lockB = new Object();

    public static void main(String[] args) {
        // 线程 1：先拿 A 再拿 B
        new Thread(() -> {
            synchronized (lockA) {
                Thread.sleep(100);
                synchronized (lockB) { } // 等线程 2 释放 B
            }
        }).start();

        // 线程 2：先拿 B 再拿 A
        new Thread(() -> {
            synchronized (lockB) {
                Thread.sleep(100);
                synchronized (lockA) { } // 等线程 1 释放 A
            }
        }).start();
        // 两个线程互相等 → 死锁！
    }
}
```

**怎么排查死锁？**
```bash
jps                     # 找到 Java 进程 ID
jstack <pid>            # 打印线程堆栈，会报告 detected deadlock
```

**避免死锁的方法：**
- 按固定顺序获取锁（所有人都先 lockA 再 lockB）
- 使用 `tryLock` 设置超时，拿不到就释放已持有的锁
- 减少锁的持有时间
- 使用更高层次的并发工具（ReentrantLock、Semaphore）

---

### 9.3 用户态和内核态

#### 正式版

操作系统为保护系统安全，将 CPU 运行状态分为**用户态**和**内核态**。用户态不能直接访问硬件资源（磁盘、网络、内存分配），需要通**系统调用**（如 `read()`、`write()`、`fork()`）请求内核态代理执行。用户态→内核态的切换开销较大（保存上下文、权限检查、复制数据）。这也是为什么 `synchronized` 重量级锁（涉及线程挂起，需要内核态操作）性能差的原因之一。

## 10. 设计模式

### 10.1 单例模式

#### 正式版

6 种写法中推荐**枚举单例**（最安全，防反射/序列化破坏）或**静态内部类**（懒加载，线程安全）。**饿汉式**：类加载时创建，简单但可能浪费资源。**懒汉式**：需要时才创建，但需要处理线程安全。**双重检查锁（DCL）**：`volatile` + `synchronized`，兼顾懒加载和效率，但代码复杂。**静态内部类**：由 JVM 保证线程安全，推荐。**枚举**：由 JVM 保证单例，防反射攻击。

---

#### 详细版

```java
// 1. 饿汉式（类加载时就创建）
public class Singleton1 {
    private static final Singleton1 INSTANCE = new Singleton1();
    private Singleton1() {}
    public static Singleton1 getInstance() { return INSTANCE; }
}
// 优点：简单，线程安全
// 缺点：类加载就创建，可能浪费资源

// 2. 懒汉式（synchronized 方法）
public class Singleton2 {
    private static Singleton2 instance;
    private Singleton2() {}
    public static synchronized Singleton2 getInstance() {
        if (instance == null) instance = new Singleton2();
        return instance;
    }
}
// 优点：懒加载
// 缺点：每次 getInstance 都要加锁，性能差

// 3. 双重检查锁（DCL）
public class Singleton3 {
    private static volatile Singleton3 instance;  // volatile！
    private Singleton3() {}
    public static Singleton3 getInstance() {
        if (instance == null) {                  // 第一次检查
            synchronized (Singleton3.class) {    // 加锁
                if (instance == null) {          // 第二次检查
                    instance = new Singleton3();
                }
            }
        }
        return instance;
    }
}
// volatile 防止指令重排序（new Singleton3() 的三步可能被重排）

// 4. 静态内部类（推荐）
public class Singleton4 {
    private Singleton4() {}
    private static class Holder {
        private static final Singleton4 INSTANCE = new Singleton4();
    }
    public static Singleton4 getInstance() {
        return Holder.INSTANCE;
    }
}
// 懒加载：Holder 类在 getInstance 调用时才被加载
// 线程安全：由 JVM 类加载机制保证

// 5. 枚举（最安全）
public enum Singleton5 {
    INSTANCE;
    public void doSomething() { }
}
// 防止反射攻击（反射不能创建枚举实例）
// 防止序列化破坏（枚举的序列化是特殊的，反序列化仍返回同一个实例）
```

## 11. 补充高频考点

### 11.1 常用 GC 收集器详解（CMS / G1）

#### 正式版

**CMS（Concurrent Mark Sweep）**：以"最短停顿时间"为目标的并发收集器。四个阶段：**初始标记**（STW，标记 GC Roots 直接关联对象）→ **并发标记**（GC Roots Tracing，与用户线程并发）→ **重新标记**（STW，修正并发标记期间变动的对象）→ **并发清除**（与用户线程并发）。优点：低延迟。缺点：**浮动垃圾**（并发清理时新产生的垃圾留到下次）、**并发模式失败**（并发标记时老年代满了，退化为 Serial Old 单线程 Full GC）、**内存碎片**（标记-清除算法天生的）。

**G1（Garbage First）**：JDK 9+ 默认。将堆划分为 2048 个 Region，每个 Region 可扮演 Eden/Survivor/Old/Humongous（大对象）。**可预测停顿**：跟踪每个 Region 的回收价值（回收能释放多少空间 + 耗时），每次选价值最高的 Region 集合回收。G1 整体是**标记-整理**算法（无碎片）。**Mixed GC**：一次 GC 同时回收新生代 + 部分老年代 Region。

---

#### 详细版

**CMS 工作流程：**

```
阶段 1：初始标记（Initial Mark）→ STW，时间短
  只标记 GC Roots 直接关联的对象
  比如栈引用、静态变量直接指向的对象

阶段 2：并发标记（Concurrent Mark）→ 与用户线程并发运行
  从 GC Roots 出发，遍历所有对象引用
  这个阶段最耗时，但因为是并发的，用户线程不暂停
  注意：此时用户线程还在运行，对象的引用关系可能变化

阶段 3：重新标记（Remark）→ STW，时间比初始标记长但比并发标记短
  修正并发标记期间用户线程运行导致变动的对象引用
  这里用到了"增量更新"算法来记录变化

阶段 4：并发清除（Concurrent Sweep）→ 与用户线程并发
  清除垃圾对象，回收空间
  因为是标记-清除算法，会产生碎片
  CMS 此时能做的就是记录空闲列表，下次分配大对象时可能找不到连续空间
```

**CMS 的并发模式失败（Concurrent Mode Failure）：**

```java
// CMS 在并发标记阶段，用户线程还在运行
// 用户线程可能会触发老年代分配（比如 new 大对象）
// 如果老年代空间不够 → "并发模式失败"
// CMS 退化为 Serial Old 单线程 Full GC（STW 时间长很多！）
// 调优方向：调大老年代大小，或提高 CMS 触发阈值
-XX:CMSInitiatingOccupancyFraction=75  // 老年代用到 75% 就启动 CMS
```

**G1 的工作流程：**

```
G1 把堆分成 2048 个 Region（每个 1MB ~ 32MB）
每个 Region 可以是 Eden、Survivor 或 Old

Young GC（一直发生）：
  选一批 Eden Region → 复制存活对象到 Survivor
  STW 但时间短（因为只处理部分 Region）

Mixed GC（老年代达到阈值后触发）：
  Step 1：初始标记（STW，很短）
  Step 2：并发标记（用户线程并发）
  Step 3：最终标记（STW）
  Step 4：筛选回收（STW，但选 Region 时考虑了性价比）
    → 哪个 Region 能释放最多空间且停顿时间最短，优先回收哪个
```

**G1 为什么能预测停顿时间？**
```java
// 启动参数设置：目标每次 GC 停顿不超过 200 毫秒
-XX:MaxGCPauseMillis=200

// G1 根据历史数据估算每个 Region 的回收耗时
// 如果预计回收 500 个 Region 需要 500ms，超出了目标
// 那就只回收 200 个 Region，控制在 200ms 内
// 这就是"可预测停顿"的含义——不一定一次回收完，但保证不超时
```

**常见 GC 面试场景题：**

**Q：你怎么判断当前 GC 有没有问题？**

```bash
# 打印 GC 日志（生产常用
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:gc.log

# 看日志中的几项关键指标：
# 1. Young GC 频率（比如 10 秒一次 → 正常，1 秒 10 次 → Eden 太小）
# 2. Full GC 频率（最好为 0，有的话要找原因）
# 3. STW 时间（G1 目标 < 200ms，CMS 目标 < 1s）
# 4. GC 后的堆占用（持续上涨 → 内存泄漏）

# 用 jstat 实时看
jstat -gcutil <pid> 1000 10  # 每秒打印一次，共 10 次
```

**Q：Full GC 频繁怎么办？**

排查步骤：
```
1. jmap -heap <pid>      查看堆使用情况
2. jmap -histo <pid>     查看各类型对象数量和大小
3. jmap -dump:format=b,file=heap.hprof <pid>  // dump 堆
4. 用 MAT（Memory Analyzer Tool）分析 dump 文件
   看大对象、泄漏嫌疑点
```

### 11.2 MySQL 锁机制（行锁 / 间隙锁 / 表锁）

#### 正式版

InnoDB 支持**行级锁**（Row Lock，锁索引记录）、**间隙锁**（Gap Lock，锁索引区间）、**临键锁**（Next-Key Lock，行锁+间隙锁的组合）和**意向锁**（Intention Lock，表级别，快速判断是否已有行锁）。InnoDB 的行锁是**锁在索引上**的，如果查询没有走索引，会升级为表锁。

---

#### 详细版

**行锁（Record Lock）：**
```sql
-- 只锁住 id=1 这一行
SELECT * FROM user WHERE id = 1 FOR UPDATE;

-- 注意：如果 user 表没有索引，或 WHERE 条件不走索引
-- InnoDB 无法确定锁哪些行 → 升级为全表扫描 + 锁所有行（相当于表锁）
```

**间隙锁（Gap Lock）：**
```sql
-- RR 级别下，范围查询会锁住不存在的"间隙"
-- 表里有 id = 1, 5, 10
SELECT * FROM user WHERE id > 2 AND id < 8 FOR UPDATE;
-- 间隙锁锁住了 (1,5) 和 (5,10) 这两个间隙
-- 其他事务不能插入 id=3 或 id=6 的新行
-- 解决幻读问题
```

**临键锁（Next-Key Lock）= 行锁 + 间隙锁：**
```sql
-- InnoDB RR 级别下，默认使用临键锁
-- 相当于同时锁住一个范围区间和这个范围内的所有行
```

**意向锁（Intention Lock）：**
```
事务 A 给某行加了行锁（X锁，排他锁）
此时表上会有一个"意向排他锁（IX）"的标记

事务 B 想给整个表加表锁
B 先检查表上的意向锁 → 发现有 IX → 知道表里有行被锁了 → 等待
如果无意向锁，B 需要遍历所有行检查是否有行锁 → 效率低
意向锁是一种"快速判断"的机制，不需要遍历
```

### 11.3 一致性哈希

#### 正式版

一致性哈希将哈希值空间组织成环形（0 ~ 2³²-1），数据通过 hash 定位到环上位置，然后"顺时针找最近的服务节点"。当节点增减时，只影响该节点在环上顺时针方向的下一个节点，**减少了大范围数据迁移**。针对数据倾斜问题引入**虚拟节点**（给每个物理节点分配多个虚拟节点，分布更均匀）。典型应用：Redis 集群分片、负载均衡。

---

#### 详细版

**传统取模哈希的问题：**
```java
// 3 台缓存服务器，用 hash(key) % 3 决定存到哪台
// 加一台变成 4 台 → hash(key) % 4 → 几乎所有 key 的映射都变了
// 缓存大面积失效 → 缓存雪崩！
```

**一致性哈希怎么做：**
```
把哈希值想象成一个环（0 → 2³²-1 → 回到 0）

服务器节点：NodeA, NodeB, NodeC
计算三个节点的 hash 放在环上

某个 key（比如 "user:1"）计算 hash 得到位置
从该位置顺时针走，遇到的第一个节点就是存储节点

假设：key 落在 NodeA 和 NodeB 之间 → 存到 NodeB

NodeA 挂了之后：
  原来存到 NodeA 的 key，顺时针走遇到 NodeB
  只影响了原来存到 NodeA 的那部分数据
  其他大部分数据不受影响

加一个 NodeD：
  只影响 NodeD 逆时针方向到上一个节点的数据
```

**虚拟节点解决数据倾斜：**
```java
// 物理节点只有 3 个，但给每个物理节点创建 100 个虚拟节点
// NodeA → NodeA_1, NodeA_2, ..., NodeA_100
// 环上有 300 个虚拟节点，每个物理节点的数据量更均衡
// NodeA 挂 → 它的 100 个虚拟节点都挂 → 数据均匀分散给其他物理节点
```

### 11.4 限流算法（令牌桶 / 漏桶）

#### 正式版

**固定窗口**：时间窗口内计数，超过阈值限流。实现简单但窗口切换时有流量尖刺。**滑动窗口**：用 ZSet 记录每次请求时间戳，看当前时间往前推一个窗口内的请求数，更平滑但内存开销大。**漏桶**：请求先进桶，桶以固定速率漏水（处理请求），桶满则新请求丢弃。**令牌桶**：匀速向桶里放令牌，请求来了拿令牌，拿不到就等或丢弃。令牌桶可以应对突发流量（桶里最多攒 N 个令牌可以瞬间用完），而漏桶强制平滑速率。

### 11.5 分布式理论（CAP / BASE）

#### 正式版

**CAP 定理**：分布式系统最多同时满足两个：**一致性**（所有节点同一时间看到的数据一致）、**可用性**（每个请求都能得到正确响应）、**分区容错性**（节点间网络断开系统仍能工作）。必须选 P（分区无法避免），所以本质是 CP 或 AP 的选择。**BASE 理论**（对 AP 的延伸）：**Basic Availability**（基本可用）、**Soft State**（软状态）、**Eventual Consistency**（最终一致性）。大多数互联网应用选择 AP + 最终一致性。

---

#### 详细版

**CAP 怎么选？**

| 系统类型 | 选择 | 例子 |
|---------|------|------|
| CP（放弃可用性） | 宁可不可用也不给错数据 | 金融交易、ZooKeeper（选举期间不可用） |
| AP（放弃一致性） | 宁可给旧数据也不能不响应 | 大部分网站、社交、商品浏览 |
| CA（放弃分区容错） | 所有节点在同一局域网 | 单体应用 |

**BASE 的具体含义：**
```
Basically Available（基本可用）：
  系统出问题时，允许损失部分功能（比如双十一降级评论功能）

Soft State（软状态）：
  允许系统存在中间状态（数据副本的同步延迟）

Eventually Consistent（最终一致性）：
  经过一段时间后，所有副本的数据最终能达成一致
  比如 DNS 修改后需要一段时间全球生效
```

### 11.6 Redis 主从 / 哨兵 / Cluster

#### 正式版

**主从复制**：一个主节点负责写，多个从节点同步数据负责读。同步过程：从节点发送 `PSYNC` → 主节点执行 `BGSAVE` 生成 RDB 发送给从节点 → 后续增量同步（replication buffer）。**哨兵模式**：在主从上加一层哨兵（Sentinel）进程，监控主节点状态，主节点挂了自动**故障转移**（从从节点中选举一个新主节点）。**Redis Cluster**：多主多从的分布式方案，数据**分片存储**（16384 个哈希槽），每个主节点负责一部分槽位。Cluster 去中心化，无中心节点，支持水平扩展。

### 11.7 Git 常用命令和流程

#### 正式版

团队协作流程：`git clone` → 切分支 `git checkout -b feat/xxx` → 开发 → `git add` → `git commit` → `git pull --rebase`（拉取最新代码并变基）→ 解决冲突 → `git push` → 提 PR。查看历史：`git log`、`git blame`。回退：`git reset`（本地）、`git revert`（远程，推荐）。暂存：`git stash`。交互式变基：`git rebase -i`（合并/修改 commit）。

---

#### 详细版

```bash
# 常用命令速查
git status                     # 查看当前状态
git log --oneline --graph     # 查看分支图

# 分支操作
git checkout -b feat-login    # 创建并切换到新分支
git branch -d feat-login      # 删除分支
git merge feat-login          # 合并分支

# 拉取代码（推荐 rebase 避免多余的 merge commit）
git pull --rebase origin main

# 撤销
git add 加错了 → git reset HEAD <file>   # 取消暂存
git commit 但没 push → git commit --amend # 修正 commit 信息
git commit 但错了 → git reset --soft HEAD~1  # 撤销 commit 保留修改
                     git reset --hard HEAD~1  # 撤销 commit 丢弃修改（慎用）

# 临时切换分支
git stash save "WIP: xxx"     # 暂存当前工作
git stash pop                 # 恢复暂存

# 查看谁写了这段代码
git blame src/main/java/com/xxx/UserService.java

# 交互式 rebase（合并多个 commit）
git rebase -i HEAD~3          # 合并最近 3 个 commit
```

### 11.9 排序算法（快排 / 归并 / 堆排）

#### 正式版

**快排**：选基准（pivot），比基准小的放左边，大的放右边，递归。平均 O(n log n)，最坏 O(n²)，不稳定。空间 O(log n)（递归栈）。**归并**：分治思想，先拆成单个元素，再两两合并成有序数组。O(n log n) 稳定，空间 O(n)。**堆排**：建大顶堆，堆顶与末尾交换，堆大小减 1 后调整。O(n log n)，不稳定，空间 O(1)。面试手写频率：快排 > 归并 > 堆排。

#### 详细版

```java
// 快速排序（面试手写频率最高）
public void quickSort(int[] arr, int left, int right) {
    if (left >= right) return;
    int pivot = partition(arr, left, right);
    quickSort(arr, left, pivot - 1);
    quickSort(arr, pivot + 1, right);
}

private int partition(int[] arr, int left, int right) {
    int pivot = arr[right];        // 选最右边为基准
    int i = left;                  // i 指向比 pivot 大的第一个位置
    for (int j = left; j < right; j++) {
        if (arr[j] < pivot) {      // 小于基准的交换到左边
            swap(arr, i, j);
            i++;
        }
    }
    swap(arr, i, right);           // 把基准放到中间
    return i;
}

private void swap(int[] arr, int i, int j) {
    int tmp = arr[i];
    arr[i] = arr[j];
    arr[j] = tmp;
}
```

```java
// 归并排序
public void mergeSort(int[] arr, int left, int right) {
    if (left >= right) return;
    int mid = left + (right - left) / 2;
    mergeSort(arr, left, mid);
    mergeSort(arr, mid + 1, right);
    merge(arr, left, mid, right);
}

private void merge(int[] arr, int left, int mid, int right) {
    int[] temp = new int[right - left + 1];
    int i = left, j = mid + 1, k = 0;
    while (i <= mid && j <= right) {
        temp[k++] = arr[i] <= arr[j] ? arr[i++] : arr[j++];
    }
    while (i <= mid) temp[k++] = arr[i++];
    while (j <= right) temp[k++] = arr[j++];
    System.arraycopy(temp, 0, arr, left, temp.length);
}
```

### 11.10 时间复杂度速记

| 复杂度 | 含义 | 代表算法/操作 |
|--------|------|-------------|
| O(1) | 常数时间 | 数组随机访问、HashMap get |
| O(log n) | 对数时间 | 二分查找、红黑树查找、B+ 树查找 |
| O(n) | 线性时间 | 遍历、LinkedList 随机访问 |
| O(n log n) | 线性对数 | 快排/归并/堆排 |
| O(n²) | 平方时间 | 冒泡/选择/插入排序、两层循环 |
| O(2ⁿ) | 指数时间 | 递归斐波那契（没优化） |
